# DECISIONS — журнал решений в неопределённостях

Формат: **[Фаза] решение** — почему, где в коде. Документы (ТЗ v4, DEV_GUIDE) —
единственный источник правды; здесь фиксируются только те развилки, которые
документы не решают напрямую.

---

## Фаза 0

- **DB-адаптер: try/catch require('better-sqlite3'), fallback node:sqlite** —
  ТЗ §20.8 требует именно эту пару без деталей выбора момента проверки. Выбрал
  ленивую загрузку в `db.js:pickDriver()` (проверяется при первом `openDb()`,
  не при импорте модуля) — чтобы одна недоступная сборка не валила импорт
  `journal.js` целиком. better-sqlite3 на этой машине собрался нормально
  (Windows, Node 22.23.1) — оба драйвера проверены `evals/race.js`.
- **journal.db: WAL + busy_timeout=5000 на обоих драйверах** — ТЗ требует
  BEGIN IMMEDIATE для atomic capture; WAL снижает вероятность SQLITE_BUSY при
  параллельных воркерах (Фаза 6), busy_timeout — подстраховка вместо ручного
  retry-цикла на уровне журнала.
- **root_spec: ключ project_id (default 'default'), а не глобальный singleton**
  — ТЗ §4 описывает root_spec как факт, не даёт схему. Сделал таблицу с PK
  project_id, чтобы не блокировать многопроектность в будущем без миграции,
  при этом v1 всегда использует 'default'. `journal.js: setRootSpec/getRootSpec`.
- **skills.js: hallucinationGuard — regex по чужим расширениям + проверка,
  что файловые токены урока входят в touches_files задачи** — §4 описывает
  правило словами ("ссылается только на реальные файлы задачи"), конкретный
  алгоритм не дан. Реализовал как чистую функцию `skills.js:hallucinationGuard`,
  используется из `agents/skill.js` (Фаза 3) и из `writeSkill()` напрямую.
- **io.js: sanitizeLine здесь, а не в talk.js** — шрам 32 про санитизацию
  ввода сеанса привязан к слою чтения строк, а не к конкретной точке входа;
  так переиспользуется всеми будущими REPL (talk.js, resume.js).
- **mock.js в Фазе 0 — только каркас (MOCK_HANDLERS пуст, registerMockHandler
  экспортирован)** — агентов ещё нет, реальные ветки появляются вместе с
  ролями (Фаза 1: coder/tester; Фаза 3: advisor/router/architect/skill;
  Фаза 5.5: live_in), как явно указано в DEV_GUIDE §2.
- **package.json: better-sqlite3 в optionalDependencies, не dependencies** —
  чтобы `npm install` не падал целиком на машинах без toolchain для нативной
  сборки; адаптер и так рассчитан на отсутствие пакета.
- Два маленьких реальных вызова `claude -p --model haiku ...` (~$0.004
  суммарно) сделаны ДО фазы 1 для проверки недокументированных флагов CLI
  (`--system-prompt-file`, полный набор изоляции) — эти флаги не описаны в
  `claude -p --help`, только упомянуты в тексте `--bare`. Без проверки нельзя
  было доверять контракту шлюза. Дальнейшие реальные прогоны отложены в
  RUNBOOK.md по вашей просьбе.

## Фаза 1

- **Windows: `spawn('claude', args, {shell:true})` + ручное
  `JSON.stringify()`-квотирование каждого аргумента** — обнаружен реальный баг
  при сборке шлюза: прямой `spawn('claude', ...)` без shell даёт `ENOENT`
  (Windows не резолвит `.cmd`-шим по PATH без шелла), `spawn('claude.cmd', ...)`
  без shell даёт `EINVAL` (известная особенность Node на Windows для
  .bat/.cmd), а `shell:true` с аргументами-путями, содержащими пробел, обрезает
  аргумент по первому пробелу без ручного квотирования. Решение проверено
  вживую (см. `gateway.js:buildSpawnArgs`). На POSIX — обычный `spawn` без
  shell, без кавычек (кавычки на POSIX испортили бы аргумент, там шелл не
  участвует). Это и есть требуемая инвариантом §20.6 «Windows-ветка для
  claude.cmd», просто конкретный механизм не был описан в документах.
- **checkRunner: shell-обвязка критериев — всегда `bash -c`, включая Windows**
  — `bash` из Git for Windows резолвится напрямую (это настоящий `.exe`, не
  `.cmd`-шим), поэтому кросс-платформенно работает без shell:true и без
  Windows-квотирования. Соответствует прямому указанию DEV_GUIDE §0
  («bash-зависимости checkRunner документированы: Windows = Git Bash»).
- **workspace/package.json создаётся координатором автоматически, БЕЗ поля
  "type"** — реальный прогон MOCK-сценария калькулятора упал: Node ищет
  ближайший package.json вверх по дереву, находил корневой package.json LOOM
  (`"type":"module"`) и трактовал `calc.js` продукта как ESM, из-за чего
  `module.exports` кидал `ReferenceError`. Ни ТЗ, ни DEV_GUIDE это не
  упоминают — фикс в `coordinator.js:ensureWorkspacePackageJson`. LOOM не
  должен навязывать продукту клиента свою модульную систему.
- **Auto-commit safety net в варианте А — безусловный git-коммит харнессом
  после каждого хода кодера, а не «страховка на случай, если кодер забыл
  git_commit»** — в JSON-варианте А у кодера физически нет git-инструмента
  (нет tool_use), поэтому «забыть закоммитить» структурно невозможно; сам
  факт того, что коммитит харнесс, а не модель — и есть реализация принципа
  шрама 4 в этом варианте (§7: полноценный `submit_result`/git_commit —
  спящая спецификация варианта Б). `coordinator.js:autoCommit`.
- **Критерий = `{cmd, expect}` (expect опционален); JS-инлайн распознаётся
  строго по форме `node -e "<код>"` / `node --eval '<код>'` целиком** —
  DEV_GUIDE даёт контракт `{cmd, expect}` без грамматики извлечения. Регэксп
  матчит ТОЛЬКО команду, целиком состоящую из `node -e "..."`; остальные
  команды (`node --check file.js`, готовый `.cjs`, `curl`) идут в bash как
  есть — извлекать нечего, шрам 7 неприменим. `checkRunner.js:NODE_EVAL_RE`.
- **cli.js принимает свободный текст, `--file task.json` или флаги
  `--title/--spec/--criteria/--role/--type/--budget`** — DEV_GUIDE говорит
  только «задача задаётся руками в коде/аргументом», не давая формат. Выбрал
  три пути ввода: быстрый смоук (позиционный текст, без критерия — уйдёт в
  blocked_needs_human, что честно), файловый (для содержательных ручных
  прогонов вроде DoD sum.js — см. RUNBOOK.md), и флаговый.
- **mock.js: различение задач сценария калькулятора по содержимому
  промпта (`/calc\.js/`, `/index\.html/`, наличие секции «Отчёт тестера»),
  а не по глобальному счётчику вызовов** — счётчик вызовов ролей общий на
  процесс и не привязан к конкретной задаче; текстовое распознавание
  устойчиво к порядку обработки очереди координатором.
- **gateway.js: cost_usd берётся из `total_cost_usd` в JSON-конверте
  claude-cli напрямую** (подтверждено реальным вызовом) — резервная формула
  по `TIER_PRICES_USD_PER_1M` используется только если поле вдруг отсутствует.
  Это точнее и проще условной таблицы цен, о которой пишет §13 ТЗ.

## Фаза 2

- **«Ход» для правила дельты (§6.1) = номер попытки (task.attempts) ИМЕННО
  этой задачи**, не глобальный счётчик вызовов кодера. Ход 1 и ход 2 —
  полный снимок, ход 3+ — added/changed/removed относительно снимка,
  показанного на предыдущем ходу ЭТОЙ ЖЕ задачи. Состояние — in-memory
  `Map(task_id → snapshot)` в `agents/coder.js`; при рестарте процесса между
  попытками откатываемся к полному снимку — деградация в сторону БОЛЬШИХ,
  но корректных данных, не баг. Очищается по завершении задачи
  (`clearEyesState`) координатором, чтобы не течь памятью на длинных прогонах.
- **claimNext() больше НЕ инкрементирует attempts; инкремент — отдельная
  `incrementAttempts()`, вызываемая координатором ПОСЛЕ проверки бюджета,
  ПЕРЕД вызовом кодера.** Обнаружено при живой прогонке evals/budget.js:
  изначально attempts инкрементился прямо в захвате (Фаза 1), из-за чего
  задача, остановленная бюджетом ДО первого реального вызова кодера,
  показывала attempts=2 вместо 1 — захват без реальной попытки тратить деньги
  не должен считаться попыткой. Это меняет семантику `claimNext()` из Фазы 1
  (правка сделана в её же файле, не поверх — гонка по-прежнему защищена той
  же BEGIN IMMEDIATE транзакцией, `evals/race.js` зелёный после правки).
- **evals/budget.js: MOCK_CALL_COST_USD = 0.01 (config.js)** — в MOCK-режиме
  каждый вызов роли раньше логировался с cost_usd=0, что делает бюджет-стоп
  непроверяемым без реальных денег. Синтетическая стоимость 0.01 — тот самый
  «игрушечный лимит» из DoD DEV_GUIDE §4, позволяет проверить остановку
  детерминированно и бесплатно.
- **mock.js: coder-хендлер для задач вне сценария калькулятора возвращает
  файл-заглушку вместо throw** — чтобы evals/budget.js (и будущие MOCK-эвалы)
  могли переиспользовать общий MOCK_HANDLERS.coder, не расширяя сам
  сценарий-эталон калькулятора.
- **Проверка prompt-кэша и дельты tokens_in по avg() — отложены в RUNBOOK.md**
  как реальные прогоны (нужны настоящие вызовы claude-cli, MOCK всегда даёт
  tokens_in=0). Механизм логирования (cache_read/cache_creation в
  events.payload) уже на месте с Фазы 1 — RUNBOOK описывает точную команду
  проверки и SQL.

## Фаза 3

- **`\b` в JavaScript-регэкспах не распознаёт кириллицу как `\w`** — реальный
  баг, найденный дважды за эту фазу: (1) в MOCK-роутере классификация tweak
  по `/^поменяй\b/i` не матчилась на «поменяй вывод...» (граница
  буква→пробел не считается word boundary, когда буква кириллическая);
  (2) в `agents/advisor.js:TECH_QUESTION_RE` — «px/мс» не ловилось тем же
  образом, из-за чего код-уровневый guard шрама 15 фактически не работал бы
  на реальных технических вопросах с кириллицей рядом. Фикс — везде, где
  термин может стоять рядом с кириллицей, использовать
  `(?<![\p{L}\d])term(?![\p{L}\d])` с флагом `u` вместо `\b`. Это НЕ описано
  ни в ТЗ, ни в DEV_GUIDE — чисто платформенная особенность JS-регэкспов,
  но напрямую бьёт по шраму 15 (советник спрашивал px), поэтому фиксирую
  здесь как критичный для доверия к guard'у баг.
- **route на ошибку парсинга роутера — буквально `'project'`, хотя это не
  входит в перечень 5 типов входа §1** — DEV_GUIDE §1 прямо пишет
  `route=project`, приоритет у DEV_GUIDE при расхождении с ТЗ. В
  `agents/router.js` 'project' — синоним «безопасный полный цикл»:
  `talk.js` направляет его в тот же `runProjectFlow` с протоколом `wish`
  (запасной, самый нейтральный протокол интервью из трёх).
- **attempts инкрементируется и для type=regression** (в отличие от
  обычных задач, где attempts = число реальных вызовов кодера) — регрессия
  вообще не вызывает кодера (§8.9), но переиспользование того же счётчика
  не создаёт путаницы (регрессия почти всегда решается за 1 «попытку» —
  один прогон tester'а) и избавляет от отдельной ветки в handleAttemptFailure.
- **Регрессионная задача зависит (blocked_by_task_id) только от ПОСЛЕДНЕЙ
  обычной задачи дерева, не от всех** — схема журнала (§4 ТЗ, дословно)
  даёт только ОДНУ колонку зависимости, полноценный fan-in (несколько
  родителей) — явно Фаза 6. Доказано (и проверено `evals/race.js`-подобной
  логикой claimNext), что при строго однопоточной обработке claimNext
  ВСЕГДА берёт самую старую pending-задачу — значит к моменту, когда
  последняя обычная задача дойдёт до done, все более ранние уже в
  терминальном статусе. Для однопоточного Фаза 3 этого достаточно и корректно;
  бросается в глаза как упрощение, которое Фаза 6 обязана заменить настоящим
  fan-in (§16).
- **advisor.js: интервью в MOCK-режиме сразу возвращает `ready`** (кроме
  специального маркера `ТЕСТ_ТЕХ_ВОПРОС` для `evals/interview.js`) — чтобы
  `npm run mock` оставался быстрым и детерминированным; сам механизм
  многоходового интервью (`continueIntake`, предохранитель
  `MAX_INTERVIEW_TURNS`) проверен структурно (используется в `talk.js`),
  но многоходовый диалог с реальными уточняющими вопросами — предмет
  реального прогона (RUNBOOK).
- **mock.js: сопоставление задач сценария — по title из "## Задача", не по
  всему тексту промпта** — обнаружен реальный баг: секция критерия
  (`## Критерий приёмки`) часто содержит имена файлов вроде `calc.js`
  (потому что сама проверка их читает), из-за чего матчинг по всему
  `userMsg` ложно триггерился. Матчинг сузили до строки title.
- **skill_writer вызывается только на статус `done`, не на
  `blocked_needs_human`** — §4 говорит «после каждой завершённой задачи»;
  трактую «завершённая» как «done», а не любой терминальный статус — урок
  из проваленной задачи менее надёжен (причина провала могла быть внешней:
  бюджет, вопрос человеку), и §4 явно называет момент «после done» через
  описание координатора.
