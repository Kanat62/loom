# LOOM — Полное техническое задание
### Автономный инженер разработки. Единая спецификация v4 (объединяет опыт v1–v2, ТЗ v3 и целевую рамку проекта)

> Документ — единственный источник правды о том, ЧТО такое LOOM и КАК он устроен.
> Каждый механизм здесь существует не потому, что «так принято», а потому что его
> отсутствие однажды убило реальный прогон — см. Реестр шрамов (§21).

---

## 0. Что такое LOOM

**LOOM — автономный инженер, а не исполнитель промптов.** Devin, Lovable, Claude Code,
Codex строят то, что сказано, пока человек сидит рядом и направляет. LOOM занимает
другую позицию: берёт ответственность за результат — думает до, во время и после,
а человек возвращается к готовому продукту, а не к диалогу.

**Формула цели:** понять глубже, чем сказано → решить до дна, а не насыпать фич →
проверить дважды (код и впечатление) → отдать продукт, почти не требующий правок.

**Принцип глубины:** «больше, чем просили» = глубже в проблему, не шире в фичи.
Каждая инициатива системы проходит фильтр: *служит ли это исходной задаче клиента?*
Wow-эффект = чувство «меня поняли глубже, чем я сам сказал», а не количество кнопок.

**Анти-цели (что LOOM сознательно НЕ делает):**
- не допрашивает «зачем» того, кто пришёл с хотелкой;
- не выдумывает данные — никогда, ни при каких обстоятельствах;
- не делает вид, что проверил то, чего проверить нельзя;
- не строит «на всякий случай» — фича без связи с задачей не создаётся.

## 1. Пять типов входа = одна жизнь проекта

Вход один (сеанс/CLI/пакет файлов), типов запроса пять. Различает их Router (§3),
будя минимально необходимый набор ролей: *не собирай совещание там, где хватит
одного взгляда*.

| Тип | Пример | Интервью советника | Кто просыпается |
|---|---|---|---|
| **question** | «что делает app.js?», «почему бот не стреляет?» | нет | только аналитик, read-only |
| **tweak** | «поменяй цвет кнопки», «почини отступ» | нет | кодер + тестер |
| **wish** (хотелка) | «хочу VTOL-дрон, напиши кодовую часть» | только «что именно должно получиться» — НИКАКИХ «зачем» | полный цикл |
| **problem** (проблема) | «растёт купля-продажа машин, не знаю какие заказывать» | диагностика: проблема → критерий успеха клиента → право оспорить решение и предложить другое | полный цикл + Исследователь |
| **spec** (ТЗ-пакет) | папка с ТЗ, доками, макетами | разбор документов, поиск дыр и противоречий в самом ТЗ, вопросы только по дырам | полный цикл |

Это не пять режимов, а одна жизнь: сданный по spec продукт живёт дальше через tweak,
tweak рождает problem, problem рождает исследование и новые задачи — всё через одну
доску одного проекта. Эскалация встроена: tweak, оказавшийся сложнее (failed или
кодер вернул вопрос), автоматически перезапускается маршрутом project.

## 2. Архитектура: два уровня, которые не смешиваются

**Нижний — умный журнал** (SQLite): единственный источник правды. Гарантирует
атомарный захват (BEGIN IMMEDIATE), зависимости задач, трейсинг run_id, бюджеты.
Не думает — защищает целостность. Переживает любую смерть процесса: доска = память.

**Верхний — роли**: советник, router, архитектор, кодер, тестер, обживатель,
исследователь, секретарь уроков. Координатор — детерминированный код без LLM,
только процесс: захват → исполнение → вердикт.

**Принцип «учёт vs ум»:** механика (кто что взял, статусы, мёржи) — код;
сложные решения — мощная модель, вызываемая разово через шлюз.

## 3. Роли

| Роль | Файл | LLM | Модель (старт) | Пишет файлы |
|---|---|---|---|---|
| advisor_intake | agents/advisor.js | да | Haiku | нет |
| advisor_analyst | ask.js / talk «question» | да | Haiku | нет |
| router | agents/router.js | да | Haiku | нет |
| architect | agents/architect.js | да | **Opus** (осн.) → Sonnet | нет (только план) |
| coder | agents/coder.js | да | Sonnet → Opus (эскалация) | да, только в worktree/workspace |
| tester | agents/tester.js | **нет** | — | нет |
| live_in (обживатель) | agents/livein.js | да | Sonnet | нет (создаёт tweak-задачи) |
| researcher | agents/researcher.js | да | Sonnet → Opus | нет (заказывает задачи) |
| skill_writer | agents/skill.js | да | Haiku | нет |
| coordinator | coordinator.js | **нет** | — | нет |

Правила ролей:
- Координатор и Архитектор не редактируют файлы — делегируют. Код пишет только Coder.
- Tester и Coordinator — чистый детерминированный код.
- Конфигурация агента = JSON (model/endpoint/tools/maxTurns); движок и инструменты общие.

**Советник — инженер-приёмщик с протоколами под тип входа.** Общие законы:
клиент описывает результат — технические решения советник принимает сам и показывает
списком «Принял инженерные решения» (Enter = согласие, текст = корректировка);
технические вопросы клиенту запрещены **на двух уровнях** — промптом И regex-фильтром
кодом (px, мс, скорости, размеры, тайминги, протоколы, форматы); мусорный вход
(«continue», <15 символов) не превращается в бриф — вежливый отказ с подсказкой /resume.
Бриф всегда содержит `engineering_defaults` — архитектор использует их дословно.
Для problem-входа советник дополнительно фиксирует: формулировку проблемы, критерий
успеха клиента, принятое решение и почему именно оно (включая случаи «вы просили X,
но вашу проблему решает Y — делаю Y, вот обоснование»).

**Router** — один вызов лёгкой модели: `{route, reason, criteria?}` (criteria обязателен
для tweak, строится по правилам §8). Парсинг безопасный: снять фенсы, JSON.parse,
при ошибке → route=project (ошибка в сторону завышения безопасна). Дополнительно
`complexity_score` (files≤1 и <20 строк → trivial-путь).

**Обживатель (второй контур тестирования, §9)** — после зелёной доски садится
за продукт как первый пользователь: Playwright-проход глазами новичка по пути
клиента; каждая шероховатость → tweak-задача на доску с обоснованием. Лимит:
один цикл обживания на сдачу (стоп-фактор от вечной полировки).

**Исследователь (§10)** — включается для problem-входов и по запросу архитектора.

## 4. Журнал

### tasks
```sql
CREATE TABLE tasks (
  id TEXT PRIMARY KEY,
  title TEXT, spec TEXT,
  criteria TEXT,                -- машинно-проверяемый критерий (§8)
  role TEXT DEFAULT 'coder',    -- coder | architect | tester | researcher
  type TEXT DEFAULT 'project',  -- project | tweak | regression | research | polish
  status TEXT DEFAULT 'pending',
  touches_files TEXT, touches_functions TEXT,   -- JSON-массивы
  blocked_by_task_id TEXT REFERENCES tasks(id),
  budget_usd REAL DEFAULT 2.0, spent_usd REAL DEFAULT 0.0,
  claimed_by TEXT, attempts INTEGER DEFAULT 0,
  feedback TEXT,                -- ТОЛЬКО последний отчёт тестера, не история
  question TEXT, created_at INTEGER
);
```
Статусы: `pending → claimed → done | failed | blocked_needs_human | merge_pending → merged`.
Захват: `UPDATE ... WHERE id=? AND status='pending'` внутри BEGIN IMMEDIATE;
`claimNext(role)` фильтрует по роли и разблокированным зависимостям. Убитый процесс
оставляет claimed-задачи — `releaseStuck()` возвращает их в pending при любом входе
(resume, talk, повторный запуск).

### events (LLM Ops, §14)
```sql
CREATE TABLE events (
  id INTEGER PRIMARY KEY, run_id TEXT, task_id TEXT,
  type TEXT,                    -- thinking | action | test_result | status | usage
  agent TEXT, duration_ms INTEGER, tokens_in INTEGER, tokens_out INTEGER,
  cost_usd REAL, payload TEXT, created_at INTEGER
);
```
run_id присваивается с ПЕРВОГО вызова модели. Диагностика = SQL, не чтение логов.

### skills.db — отдельная БД
BEGIN DEFERRED (слабая строгость: перезапись урока допустима). writeSkill()
вызывается механически координатором после каждой завершённой задачи.
Фильтр галлюцинаций: урок обязан ссылаться только на реальные файлы задачи;
упоминание чужих языков/файлов (.py/.cs/…) → отброс; нет урока → SKIP, ничего
не пишется. FTS5-поиск — только после 100+ записей.

### root_spec
Исходная цель проекта хранится в журнале ВНЕ контекста агентов и перечитывается
на каждой проверке (лекарство от goal drift, §15).

## 5. Шлюз моделей

Единственная точка звонков. Контракт: `chat(role, messages) → текст`;
`chat(role, messages, {json:true}) → уже распарсенный объект` — валидация ВНУТРИ
шлюза, кривой JSON = сбой модели (лог 200 символов сырого ответа, ретрай с
напоминанием, затем следующая модель цепочки), а не смерть прогона.

- **Стартовая конфигурация (текущее решение):** единственный провайдер —
  **claude-cli**; разные роли получают разные модели через флаг `--model`
  (суффикс в имени: `claude-cli:haiku|sonnet|opus`). Тиры: лёгкие роли — Haiku,
  кодер — Sonnet с эскалацией на Opus, **архитектор — в основном Opus** (качество
  дерева и критериев определяет весь прогон, тут экономить нельзя) с фолбэком
  на Sonnet при недоступности/лимите. Пример:

  ```javascript
  export const ROLES = {
    advisor:   ["claude-cli:haiku"],
    router:    ["claude-cli:haiku"],
    architect: ["claude-cli:opus", "claude-cli:sonnet"],
    coder:     ["claude-cli:sonnet", "claude-cli:opus"],   // вверх при провале grounding
    live_in:   ["claude-cli:sonnet"],
    researcher:["claude-cli:sonnet", "claude-cli:opus"],
    cheap:     ["claude-cli:haiku"],
  };
  ```
  Следствия зафиксированы: единая точка отказа (лимит подписки останавливает всю
  машину — доска ждёт /resume, работа сессиями между окнами лимитов); HTTP-ветка
  шлюза лежит в коде выключенной и включается конфигом без правки логики.
- **Провайдеры равноправны** архитектурно: OpenAI-совместимый HTTP (NIM/DeepSeek/любой) и
  `claude-cli` (Claude Code headless по подписке). Для claude-cli: системный промпт
  через `--system-prompt-file` (аргумент CLI на Windows режется по первому \n),
  пользовательский — через STDIN; изоляция вручную: `--tools ""`,
  `--strict-mcp-config` + пустой mcp.json, `--settings` с пустыми hooks
  (`--bare` не годится: требует API-ключ, не читает подписку).
- **Цепочки фолбэков** на роль + **два круга** с паузой 60с («все недоступны» в час
  пик — не смерть). **Две причины перехода** различаются: техническая (модель упала /
  таймаут) → следующая в цепочке того же тира; содержательная (ответила, но grounding
  провален) → эскалация на модель сильнее.
- `maxRetries: 0` в SDK — вся retry-логика только в шлюзе (дефолт SDK молча утраивает
  таймауты). Таймаут вызова 300с (180с убивал честно думающий DeepSeek).
- `MODEL_LIMITATIONS` в конфиге (пример: llama — один tool_call за ход → пропуск
  модели, если история многовызовная). THINKING per-role: reasoning выключен для
  advisor/router/cheap (минуты на «задай 3 вопроса» — расточительство).
- **Кэширование промптов:** системный промпт — статичный файл, побайтово идентичный
  между вызовами; вся динамика — в user message. Проверка: cache_read_input_tokens > 0
  на втором вызове. Лог usage (in/out/cache) каждого вызова → events.

## 6. Context Engineering

1. **Глаза кодера — дельта, не снимок.** Ход 2 — полный snapshot, ходы 3+ — только
   added/removed/changed. (v2 слал весь workspace: 34К токенов на ход.)
2. **Фильтр мусора в глазах:** файлы `.min.*`, файлы >50КБ, бинарники → строка-заглушка
   `"[библиотека chart.min.js, 180КБ — не читается]"`. Кодеру ЗАПРЕЩЕНО записывать
   сторонние библиотеки файлами — только CDN `<script src>` (инцидент: chart.min.js
   в глазах жёг ~15К токенов каждый вызов).
3. **Релевантность:** полные тексты — только файлам из spec/touches_files задачи,
   остальным — имя + первые 30 строк.
4. **Rolling summary** истории tool_use на уровне harness: держать последние N ходов,
   старое сворачивать в сводку.
5. **feedback = только последний отчёт** тестера, не накопление истории попыток.

## 7. Loop Engineering

> **Статус:** активируется при подключении tool_use-провайдера (HTTP) или переходе
> кодера на вариант Б (инструменты внутри headless). В стартовой claude-cli-конфигурации
> действует JSON-контракт кодера (вариант А): модель возвращает
> `{"files":{...}} | {"question":"..."}`, файлы пишет harness; пункты про
> submit_result/tool_choice — спящая спецификация, лестница попыток и auto-commit
> действуют с первого дня.

- Финал хода агента — инструмент `submit_result`: `tool_choice:"required"` + сужение
  списка до одного инструмента на последнем ходу (именной forced choice часть моделей
  молча игнорирует). Прозаический ответ вместо tool_call → retry-to-next-model на
  КАЖДОМ ходу.
- Лестница попыток задачи: 1) дешёвая модель → 2) та же, чистый контекст, root_spec
  заново → 3) модель сильнее того же тира → 4) blocked_needs_human. MAX_ATTEMPTS=4.
- **Auto-commit safety net:** кодер может забыть git_commit → «ложно смёржено»;
  коммит принудительно в конце хода кодера, если тот не закоммитил сам.
- Канал «вопрос человеку»: кодер вправе вернуть question → задача в
  blocked_needs_human, вопрос человеку в интерфейс, ответ приклеивается к spec.

## 8. Свод правил критериев приёмки (сердце доверия)

Критерии пишет Архитектор ДО кода; Кодер их видит, изменить не может (журнал,
куда его инструментам хода нет). Нарушение любого правила = брак архитектора.

1. Проверяется **поведение**, не исходник: регулярки по структуре кода запрещены.
   Сайты/игры — живой браузер (Playwright) через window-объекты и DOM;
   `pathToFileURL`, не склейка `file://`+путь.
2. Клиент дал свою проверку в брифе — используется **она**, дословно.
3. **Минимумы, не точные числа** (grep -c считает строки, не совпадения). Критерий
   выполним любым правильным решением.
4. **Говорящий провал:** при неудаче печатать, какое условие не выполнено, с
   фактическими значениями (`FAIL: bots=3 moved=false`). Одинокие FAIL/ERROR
   запрещены. Отчёт читает LLM-кодер — пишется для него.
5. **Симулированные проверки запрещены:** команда не имеет права безусловно печатать
   expect. Нельзя проверить одной командой — задача режется иначе, но проверка
   не подменяется имитацией.
6. **Серверный шаблон:** `node --check` до запуска; сервер — дочерним процессом с
   пересылкой stdout/stderr (`SERVER:`-префикс) и кодом выхода; готовность —
   ретраями подключения до 3с, не фикс-паузой; kill и лимит 10с.
7. Все JS-критерии исполняются через временный **`.cjs`-файл внутри workspace**
   (checkRunner) — НИКОГДА shell-строкой (`$$` в bash портит `$$eval` до запуска
   Node; tmpdir не годится — require не дорезолвится до node_modules проекта).
8. **Предпроверка (капкан на пустышки):** каждый критерий тихо (stdio pipe, 5с)
   прогоняется на текущем workspace ДО работы кодера; проходит заранее → фиктивен →
   одна регенерация; лог одной строкой на задачу.
9. **Контракт регрессии** (класс type=regression, последняя задача каждого дерева):
   только повторный прогон УЖЕ СУЩЕСТВУЮЩИХ критериев done-задач (склейка id, не
   генерация нового кода проверки), исполняется тем же checkRunner; писать новые
   Playwright-скрипты регрессии запрещено (систематический источник битого
   синтаксиса в v2: #6, #15, #20 падали не по существу).
10. Контракт задачи целиком: измеримая цель, критерий = команда+ожидание, границы
    (что нельзя трогать), лимиты (итерации/бюджет), стоп-факторы.

## 9. Двойное тестирование

**Контур 1 — код (Checker):** тестер без LLM, отдельный чистый процесс, критерии
из журнала, таймаут, отчёты до 1000 символов (200 душили диагностику вложенных
процессов). Тесты — часть проверки, не единственная истина; финальная
ответственность — человек.

**Контур 2 — впечатление (обживатель):** после зелёной доски LOOM проходит продукт
как первый пользователь: открылось ли, понятно ли за 10 секунд, находится ли
главное действие, не ломается ли путь клиента. Итог — не вердикт, а пачка
tweak/polish-задач с обоснованиями («кнопка экспорта не видна без скролла на
375px»). Один цикл на сдачу. Именно отсюда берётся «продумано, почти без правок».

## 10. Исследователь и данные

Включается для problem-входов и по заказу архитектора («для решения нужны данные X»).

- **Ищет источники** (веб-доступ через своего провайдера), оценивает доступность,
  свежесть, покрытие.
- **Заказывает инструменты добычи конвейеру:** «нужен парсер площадки N» — обычная
  задача на доске (role=coder, criteria: «выдаёт ≥50 записей с полями марка/год/цена»).
  Рекурсия конвейера — штатный механизм.
- **Критерии данных** — свой класс проверок: свежесть (дата последней записи),
  объём (минимум N), правдоподобие диапазонов (цена авто ∉ {0, 10⁷$}).
- **Жесточайший стоп-фактор:** данных нет / добыть не удалось → честное
  blocked_needs_human с формулировкой «этих данных не существует в открытом виде»
  клиенту. Выдумать данные — худший провал системы, хуже любого падения.
- **Этика:** только публичные данные, уважение robots/ToS, без обхода защит.
- Продукт для problem-клиента формулируется честно: «LOOM собрал живые данные и дал
  инструмент видеть — решение принимает клиент», не «LOOM знает будущее».

## 11. Приём ТЗ-пакета (spec-вход)

Вход — папка/архив: ТЗ, доки, макеты, примеры. Пайплайн: разбор каждого документа →
единый root_spec → карта требований → **поиск дыр и противоречий в самом ТЗ**
(вопросы клиенту только по дырам, не по вкусам) → дерево задач с трассировкой
«требование → задача → критерий». Требование без критерия = дыра, о которой
сообщено клиенту до старта.

---

## 12. Безопасность

1. **Docker с ПЕРВОГО появления кодера с инструментами** (не фазой позже): смонтирован
   только workspace, наружу физически нельзя (инцидент: папка на рабочем столе).
   До Docker — временная решётка: замок пути (запись только внутри workspace) —
   обязательный минимум с первого дня.
2. Сеть контейнера: по умолчанию `--network none`; зависимости ставятся на этапе
   сборки образа из packages-плана архитектора; нужен пакет посреди задачи →
   статус «нужен пакет» → пересборка. Позже — egress-allowlist (registry.npmjs.org).
   Лимиты: `--memory`, `--cpus`, `--pids-limit`.
3. **Maker не трогает Checker:** allowlist (не blocklist) на защищённые файлы;
   grader в отдельном процессе (инцидент: подмена assert).
4. **gitGuard:** физическая защита репозитория самого LOOM от stray-веток агентов.
5. **Политика секретов:** ключи живут только в `.env` вне репо; в документах, ТЗ,
   скриншотах, коммитах — никогда (два реальных инцидента: nvapi-ключ на скриншоте,
   GitHub-ключи в теле ТЗ v3). Утёк — немедленный отзыв.
6. Инструменты агента не имеют доступа к journal.db и prompts/ (территория Checker).

## 13. Бюджеты и тиры

`budget_usd`/`spent_usd` на задаче: фактическая стоимость каждого вызова
прибавляется; проверка ДО следующего вызова; превышение → blocked_needs_human.
Второй уровень — бюджет проекта (координатор). Тиры: лёгкая модель по умолчанию,
дорогая — только когда дешёвая не справилась (лестница §7). Подписочный провайдер
(claude-cli) учитывается условной ценой для сопоставимости метрик.

## 14. LLM Ops / Evals

- **Уровень 1:** run_id-трейс с первого вызова; любой инцидент = SQL-запрос.
- **Уровень 2:** агрегаты поверх events: стоимость по ролям, % задач с первой
  попытки, файлы-конфликтогены, тренд стоимости задачи, латентность моделей.
- **Уровень 3:** LLM-as-judge (лёгкая модель) — оценка DAG архитектора после прогона
  (атомарность, дубли, реалистичность touches_files) → eval_score в events.
- **Eval интервью** (урок регрессии советника): банк эталонных желаний → прогон →
  автопроверка «нет технических вопросов клиенту» (лёгкой моделью: «есть ли в вопросе
  числа/алгоритмы/форматы?»). Промпт — тоже код, и ему нужны тесты.

---

## 15. Дрейф — три типа и лекарства

- **Context drift** (забывает начало к 15-му ходу) → §6: дельты + rolling summary.
- **Goal drift** (уезжает от ТЗ) → root_spec вне контекста + pre-work confirmation +
  перечитывание цели на каждой проверке.
- **Coordination drift** (параллельные агенты к несовместимым целям) →
  declares_interface (tsc --noEmit как level-0) + touches_files с реактивной
  перепроверкой.

---

## 16. Параллельность (после стабильного однопоточного режима)

- CODER_WORKERS=N, каждый в своём git worktree. Правило масштаба: параллелятся
  прежде всего задачи из разных зон дерева; 2–3 воркера — норма, 5 — потолок
  (структура зависимостей, а не железо — узкое место).
- **Sequential mergeQueue:** done → merge_pending → merge в main → rebase остальных.
- **Entity-level claiming:** touches_functions; два агента в одном файле допустимы
  на разных функциях (tree-sitter merge driver).
- **Конфликт после rebase:** авто-мёрж непересекающихся → LLM-reconciler (обе ветки +
  задачи; ~94% решается выбором существующего варианта, лёгкой моделью) → эскалация.
- **Fan-in:** failed блокирует done-ветки? да → проект blocked_needs_human;
  нет → продолжать, failed в retry-очередь.

---

## 17. Сдача: отчёт консультанта

Сдаётся не «вот файлы», а позиция:
«Твоя задача/проблема была — …; я принял решения — … и вот почему; данные взяты
оттуда-то (живые, обновляются так-то); проверено так-то (оба контура); вот что
требует твоего решения; вот что я бы сделал следующим шагом.»
Для wish-входа блок «проблема» опускается — только «что получилось и как проверено».

## 18. Maintenance после сдачи

Правки и развитие идут той же доской того же проекта (tweak/project). Автономные
изменения живого прода — только после многократно пройденного уровня Checker:
PR → CI в чистом окружении (не верить локальному вердикту агента) → staging +
health-период → прод; rollback одной командой; rate-limit автономных мёржей в сутки;
мониторинг прода порождает задачи regression_detected в журнал.

---

## 19. Диагностическое правило четырёх уровней

Баг повторяется после правки промпта → это НЕ Prompt-проблема. Уровни:
**Prompt** (что написано) → **Context** (что модель видит) → **Harness**
(инструменты, изоляция, fallback) → **Loop** (когда/как завершается цикл).
Чинить строго на правильном уровне. Канонический пример: submit_result не вызывался —
промпт не помог, решилось на Loop (tool_choice required + сужение списка).

## 20. Инварианты среды (нарушение = баг)

1. `--env-file=.env` во ВСЕХ npm-скриптах (не пакет dotenv).
2. `maxRetries: 0` в SDK.
3. Проект вне iCloud/облачной синхронизации (`~/dev/loom`).
4. Все JS-критерии — через `.cjs`-файл, никогда shell-строкой.
5. run_id в events с первого вызова модели.
6. Системный промпт claude-cli — только файлом; user — только STDIN
   (Windows режет аргумент по \n).
7. Секреты — только `.env`, никогда в документах/репо/скриншотах.
8. Кроссплатформенность явная: пути через node:path; bash-зависимости checkRunner
   документированы (Windows = Git Bash); БД — better-sqlite3 за тонким адаптером
   journal-интерфейса, fallback node:sqlite (нативная сборка на Windows падала).
9. Ввод сеанса санитизируется (инцидент: склейка терминального буфера в title задачи).
10. MOCK-ветка каждой фичи — ПОСЛЕ проверки реального пути, не до.

## 21. Реестр шрамов (обязателен к прочтению новыми разработчиками)

Каждая строка — реальный инцидент → механизм. Удалять механизм можно только вместе
с объяснением, почему шрам больше не актуален.

1. `.env` не грузился → инвариант --env-file
2. maxRetries:2 SDK утраивал таймауты → maxRetries:0, retry в шлюзе
3. Прозаический ответ принят как финальный → tool_choice required + retry на каждом ходу
4. Кодер пропускал git_commit → auto-commit safety net
5. Stray-ветка на репо LOOM → gitGuard
6. iCloud dataless node_modules → проект вне облачных папок
7. `$$` в shell-критериях ломал $$eval до Node → checkRunner, .cjs в workspace
8. llama: один tool_call за ход → MODEL_LIMITATIONS
9. currentFiles слал весь workspace (34К/ход) → дельта
10. Вымышленные ID моделей (404) → сверка с /v1/models, никакого хардкода по памяти
11. Кривой JSON модели убивал прогон → валидация внутри шлюза, не-JSON = сбой модели
12. Таймаут 180с убивал думающий DeepSeek (514с ответ) → 300с
13. «Все модели недоступны» в час пик → второй круг через 60с, длинные цепочки
14. Reasoning на «задай 3 вопроса» (минуты) → THINKING off для лёгких ролей
15. Советник спрашивал клиента «скорость в px» → product/engineering split + regex-фильтр КОДОМ
16. Бриф-галлюцинация из «continue» → guard мусорного входа + /resume
17. Слепой кодер переписал рабочую игру с нуля → «глаза» (текущие файлы в контексте)
18. module.exports убил браузерную игру → тип кода по задаче в промпте кодера
19. Критерий-пустышка «Simulated check» → запрет + предпроверка с регенерацией
20. Немой «BOTS_FAIL» — кодер чинил вслепую → говорящие провалы, отчёты 1000 символов
21. grep -c и точные числа валили правильный код → правило минимумов
22. Некому ставить ws/express → packages-план + npm install до кодера
23. stderr упавшего сервера терялся в критерии → серверный шаблон (SERVER:, node --check)
24. Уроки про Unity/Python в JS-проекте → фильтр галлюцинаций + SKIP
25. Задачи навечно в doing после смерти процесса → releaseStuck при каждом входе
26. Правка без отката похоронила проект → .history-бэкап перед каждым изменением + git
27. readline терял опережающий ввод → очередь строк в io
28. Windows обрезал CLI-аргумент по \n (промпт = одно слово) → system-prompt файлом, user через STDIN
29. --bare требовал API-ключ, игнорируя подписку → ручная изоляция headless (tools "", strict-mcp, settings)
30. chart.min.js в «глазах» жёг ~15К токенов/вызов → фильтр .min/размера + правило CDN
31. Регрессионные критерии систематически битые (#6, #15, #20) → контракт регрессии: только повтор существующих критериев
32. Терминальный мусор склеился в title задачи → санитизация ввода сеанса
33. Секреты в скриншоте и в теле ТЗ → политика секретов (§12.5)

## 24. MOCK-режим

Каждый механизм имеет MOCK-ветку (после реального пути): детерминированные ответы
ролей, сценарий с нарочной ошибкой кодера в первой попытке (демонстрация цикла
поймал→отчёт→починил), маркеры маршрутов router. `MOCK=1` — полный прогон без
единого вызова модели; обязательный смоук перед любым реальным запуском и после
любой правки машины.
