## 3. Фаза 2 — Context Engineering (prompt-кэш)

**[РЕАЛЬНЫЙ ПРОГОН]** Два вызова подряд одной и той же роли с ОДИНАКОВЫМ
системным промптом должны показать `cache_read_input_tokens > 0` на втором:
```bash
node --env-file=.env --no-warnings -e "
import('./gateway.js').then(async (g) => {
  const messages = [
    { role: 'system', content: 'Ты тестовый ассистент. Всегда отвечай ровно: OK' },
    { role: 'user', content: 'ping' },
  ];
  await g.chat('cheap', messages, { json: false, run_id: 'cache-check-1' });
  await g.chat('cheap', messages, { json: false, run_id: 'cache-check-2' });
  console.log('готово — смотри events ниже');
});
"
node --env-file=.env --no-warnings -e "
import('./journal.js').then(j => {
  for (const e of j.listEvents({ type: 'usage' })) console.log(e.run_id, JSON.parse(e.payload).cache_read);
});
"
```
Ожидание: у события с `run_id='cache-check-2'` поле `cache_read` больше 0
(первый вызов создаёт кэш — `cache_creation` > 0 — второй его читает).
Системный промпт короткий, кэш у Anthropic обычно требует ~1024+ токенов —
если `cache_read` всё ещё 0, увеличьте текст системного промпта в этом
тесте (не критично для боевых промптов — `coder.md`/`architect.md` заведомо
крупнее порога).

Сравнение токенов по агентам (агрегат §14 ТЗ):
```bash
node --env-file=.env --no-warnings -e "
import('./journal.js').then(j => {
  // грубое агрегирование без raw SQL — адаптер намеренно не отдаёт сырой доступ
  const events = j.listEvents({ type: 'usage' });
  const byAgent = {};
  for (const e of events) {
    byAgent[e.agent] ??= { n: 0, sumIn: 0 };
    byAgent[e.agent].n++;
    byAgent[e.agent].sumIn += e.tokens_in || 0;
  }
  for (const [agent, s] of Object.entries(byAgent)) console.log(agent, 'avg tokens_in =', (s.sumIn / s.n).toFixed(0));
});
"
```

Бюджет-стоп на реальном лимите уже проверен в MOCK (`npm run budget-eval`) —
механизм идентичен для реальных вызовов, отдельного реального прогона не
требует.

## 5. Фаза 4 — Docker-песочница

Демон Docker в среде сборки был недоступен (`docker info` падал) — весь
код написан и синтаксически проверен, но НИ РАЗУ не прогнан на реальном
контейнере. Проверьте здесь.

1. Запустите Docker Desktop, дождитесь `docker info` без ошибок.
2. Повторите любой прогон из раздела 4 (`npm run talk` + задача). В логе
   координатора должно появиться:
   ```
   [coordinator] Docker-песочница активна для этого workspace (--network none, лимиты).
   ```
   Если вместо этого — `Docker доступен, но сборка образа не удалась: ...`,
   смотрите текст ошибки (обычно — нет `workspace/package.json`, но
   координатор создаёт его сам при первом заходе, так что это не должно
   всплывать в норме).
3. **rm -rf / не трогает хост.** ⚠️ Выполняйте ЭТОТ шаг только после того,
   как шаг 2 подтвердил `Docker-песочница активна` — без Docker команда
   `rm -rf /` пойдёт по локальному bash-фолбэку и НЕ будет изолирована.
   ```bash
   node --env-file=.env --no-warnings evals/manual/docker-isolation-task.js
   npm run resume
   ```
   (`evals/manual/docker-isolation-task.js` создаёт задачу type=regression —
   coordinator пропускает вызов кодера для регрессии и сразу отдаёт критерий
   tester/checkRunner, ровно то звено, что уезжает в контейнер — реальный
   вызов модели не тратится.) Ожидание: команда отработает ВНУТРИ контейнера
   (файловая система контейнера эфемерна и не пересекается с хостом — это
   стандартная изоляция Docker, наши флаги `--network none --memory 2g
   --cpus 2 --pids-limit 256` дополнительно ограничивают ресурсы и сеть, но
   изоляцию файловой системы даёт сам факт контейнеризации). После прогона
   убедитесь, что `C:\Users\USER\Desktop\dev\loom` и остальной хост целы.
4. Путь замка (`../evil.js`) уже проверен в MOCK и не требует Docker
   (харнесс-уровневая защита работает независимо, см. `npm run mock`,
   сценарий "path-lock").

## 6. Фаза 5.5 — обживание и отчёт консультанта

Продолжение прогона из раздела 4 (кофейня) — обживание запускается
АВТОМАТИЧЕСКИ координатором `talk.js` после того, как всё дерево (включая
регрессию) дошло до `done`, ОДИН раз. Смотрите хвост того же прогона:
```
[live_in] обживаю продукт как первый пользователь...
[live_in] найдено шероховатостей: N, создано polish-задач: N
--- Отчёт консультанта ---
...
```

**Опционально — живой браузер вместо статического разбора HTML:**
```bash
npm install playwright
npx playwright install chromium
```
Без этого `live_in` использует честный fallback (читает HTML напрямую,
помечает данные `live:false` в промпте) — работает, но видит меньше
(не проверяет реальный рендер/видимость элементов на 375px).

**DoD-чеклист отчёта** (проверьте глазами вывод из раздела 4):
- [ ] есть блок «принятые решения» с «почему» для каждого
- [ ] упомянуты оба контура проверки (код + впечатление)
- [ ] явно сказано, есть ли что-то, требующее вашего решения (или что нет)
- [ ] для wish-входа НЕТ блока «проблема» (см. `prompts/advisor.report.md`)

## Ф5. Фаза 5 — публикация в GitHub (П§3)

**[РЕАЛЬНЫЙ ПРОГОН]**
1. `gh auth status` — без ошибок (если `gh` не установлен/не залогинен,
   пропустите: прогон и без этого раздела корректно печатает «репозиторий
   не создан», это тоже проверенное поведение).
2. В `.env`: `GITHUB_PUSH=1`.
3. Любой project-прогон, доводящий дерево до `done` + обживания + отчёта
   консультанта (раздел 4/6 выше — любое желание подойдёт). Сразу после
   строки «--- Отчёт консультанта ---» должно появиться:
   ```
   [github] опубликовано: https://github.com/<ваш-аккаунт>/<слаг-имени-желания>
   ```
4. Откройте URL — приватный репозиторий с полной историей коммитов по
   задачам (`loom: done task_...`, `loom: coder attempt N on task_...`,
   сообщения не переписаны — Фаза 5 только публикует то, что уже
   закоммитил `autoCommit`).
5. Повторный прогон в ТОМ ЖЕ workspace (remote уже есть) должен пойти по
   ветке `git push origin HEAD`, не пытаться создать репозиторий второй раз.
6. Уберите `GITHUB_PUSH=1` из `.env`, если не хотите публиковать дальнейшие
   учебные прогоны автоматически.
