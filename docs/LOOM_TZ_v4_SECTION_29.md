## 29.0. Инцидент (реальный, подтверждён на живом прогоне)

Во время прогона проекта «Пульс кофейни» `/resume` начал выполнять задачи
ДРУГОГО, старого проекта (маркетплейс машин: `cars.ts`, `CarCard.tsx`,
vite/React-стек, «панель фильтра»; плюс задача `devkg-jobs-scraper` третьего
проекта про вакансии) — жёг бюджет и попытки на чужом дереве, а `/status`
показывал сводные счётчики по всей базе. Разбор кода показал четыре
самостоятельных дыры одной природы:

1. `bin/talk.js` во всех вызовах (`getRootSpec/setRootSpec/addTask/...`)
   жёстко передаёт `'default'` — колонка `tasks.project_id` в схеме
   существует, но фактически все проекты живут под одним id;
2. `claimNext` не имеет фильтра по проекту вовсе — SQL выбирает любую
   pending-задачу роли по всей базе; координатор одного проекта исполняет
   чужие деревья;
3. `root_spec` под ключом `'default'` НАКОПИТЕЛЬНО мёржится между
   несвязанными проектами (`mergeRootSpec`) — советник/архитектор видят
   слитую спеку всех продуктов сразу; это тишайшая и самая ядовитая часть:
   она портит не только очередь, но и СМЫСЛ контекста всех ролей;
4. все продукты пишут в один общий `workspace/` — файлы разных продуктов
   лежат вперемешку, глаза кодера и live_in видят чужой код, autoCommit
   мешает историю продуктов в один git.

Урок для реестра шрамов (§21, шрам 35): «колонка в схеме — не изоляция;
изоляция — это фильтр в КАЖДОЙ точке чтения и записи». Механизм — этот раздел.

## 29.1. Модель: проект как первоклассная сущность

### Идентичность

- `project_id` = слаг из summary брифа + короткий хеш (`coffee-pulse-3f2a`);
  генерируется harness'ом детерминированно, санитизация — существующая
  (правило слагов, кириллица транслитерируется или отбрасывается — как уже
  сделано для имени GitHub-репозитория в Фазе 5, переиспользовать ту же
  функцию);
- новая таблица-реестр:
  ```sql
  CREATE TABLE IF NOT EXISTS projects (
    id TEXT PRIMARY KEY,
    title TEXT,
    workspace_dir TEXT,
    status TEXT DEFAULT 'active',   -- active | archived | done
    created_at INTEGER,
    last_active_at INTEGER
  );
  ```
- активный проект сессии talk хранится в состоянии сессии (переменная), при
  старте talk выбирается: единственный active → он; несколько → печатается
  список с номерами и статусом задач, человек выбирает; ноль → «нет активных
  проектов, опишите задачу»; команды: `/projects` (список со счётчиками
  задач по статусам), `/project <id|номер>` (переключить).

### Когда рождается новый проект

Детерминированное правило в talk.js (НЕ отдано модели): маршрутизатор дал
`wish`/`problem`/`spec` И (активного проекта нет ИЛИ человек явно отвечает
«это новый продукт» на один короткий вопрос harness'а «это про текущий
проект <title> или новый продукт?») → создаётся новая запись projects +
СВОЙ workspace. `tweak`/`question` всегда адресуются активному проекту —
если активного нет, honest-ошибка «сначала выберите проект: /projects».

### Workspace на проект

- `WORKSPACE` из конфига становится КОРНЕМ: продукт живёт в
  `<WORKSPACE>/<project_id>/`; путь пишется в `projects.workspace_dir`;
- все существующие функции уже принимают `workspaceDir` параметром
  (кодер/тестер/checkRunner/live_in/worktrees/github) — менять сигнатуры не
  нужно, меняется только ТО, ЧТО в них передают talk и координатор;
- замок пути (§12) параметризуется base'ом каталога проекта — попытка
  задачи проекта А написать в каталог проекта Б блокируется тем же
  механизмом, что `../`;
- git: `git init` на каталог проекта при создании (autoCommit уже работает
  от workspaceDir); `.worktrees` Фазы 6 — внутри каталога проекта;
- MOCK-изоляция прежняя: всё под `workspace.mock/<project_id>/`.

## 29.2. Фильтр в каждой точке (ядро фикса)

Правка `core/journal.js` + все вызыватели:

1. `claimNext(role, claimedBy, { projectId, ... })` — `projectId`
   ОБЯЗАТЕЛЕН; в SQL добавляется `AND t.project_id = ?`. Вызов без
   projectId — бросок исключения, не тихий полный скан (fail-loud: тихий
   дефолт и породил этот инцидент);
2. `releaseStuck({ projectId })` — возвращает claimed→pending только в
   пределах проекта;
3. `listTasks`, сводка `/status`, печать ⚠-блоков — только активный проект;
   отдельной строкой честно: «в других проектах: N pending, M blocked
   (/projects — список)» — чтобы чужие незавершённые деревья не исчезли из
   виду молча, но и не смешивались;
4. `getRootSpec/setRootSpec/mergeRootSpec` — только по id активного проекта;
   накопительный мёрж остаётся (это правильный механизм В ПРЕДЕЛАХ проекта),
   межпроектный мёрж становится невозможен по построению;
5. координатор (`runCoordinatorLoop`) получает projectId и передаёт его в
   каждый claimNext всех воркеров; регрессия/live_in/skill_writer — тоже;
6. события: в `events` добавить колонку `project_id` (миграция по образцу
   П§2, идемпотентно) и заполнять её во всех logEvent-вызовах, где известна
   задача/проект — SQL-диагностика (§14) должна уметь резать по проекту;
7. бюджет проекта (§13) считается по задачам проекта, не по всей базе —
   проверить агрегирующие запросы трат на фильтр project_id;
8. Кузница (§26) — НЕ по проекту: реестр tools остаётся глобальным
   (инструменты переживают проекты — это их смысл), но `tool_run`-задачи
   принадлежат проекту как обычные задачи. Явно зафиксировать в DECISIONS,
   что глобальность tools — решение, а не пропущенный фильтр.

## 29.3. Миграция загрязнённой базы (реальная база автора)

В реальном journal.db сейчас минимум три продукта под `project_id='default'`
и их файлы вперемешку в одном workspace. Автоматически расплести задачи по
проектам надёжно нельзя (по заголовкам — гадание). Честное решение:

1. миграция создаёт запись `projects('legacy', 'Legacy (до §29)',
   <старый workspace>, 'archived')` и оставляет все существующие задачи с
   `project_id='default'` как есть, но `'default'` переименовывается в
   `'legacy'` (`UPDATE tasks/root_spec SET project_id='legacy' WHERE
   project_id='default'`), чтобы имя «default» больше никогда не могло
   возникнуть как тихий дефолт;
2. `addTask` теряет `|| 'default'`-фолбэк: task без project_id — исключение;
3. архивный проект виден в `/projects`, его можно сделать активным и
   доразобрать руками (или бросить); новые проекты стартуют чисто в своих
   каталогах;
4. старый общий workspace НЕ трогается и НЕ чистится автоматически —
   файлы принадлежат человеку.

## 29.4. MOCK-покрытие (§24)

Сценарий «два проекта» без единого вызова модели:
1. создаются проект A (3 задачи) и проект B (2 задачи), у обоих pending;
2. координатор запускается для A → все claimNext вернули только задачи A
   (по events ни одного захвата B); затем для B — симметрично;
3. `/status` в контексте A показывает счётчики A + строку «в других
   проектах: 2 pending»;
4. `releaseStuck` для A при claimed-задаче в B — задача B не тронута;
5. root_spec: wish в A, затем wish в B → `getRootSpec(A)` не содержит ни
   слова из B (проверка подстрокой);
6. замок пути: задача A пишет `../<B>/evil.js` → заблокировано, событие;
7. вызов claimNext без projectId → исключение (негативный тест fail-loud);
8. миграция: фикстурная база со смешанными 'default'-задачами → после
   миграции все они 'legacy', projects-запись создана, повторный прогон
   миграции идемпотентен.
