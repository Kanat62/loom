# DECISIONS — журнал решений в неопределённостях

Формат: **[Фаза] решение** — почему, где в коде. Документы (ТЗ v4, DEV_GUIDE) —
единственный источник правды; здесь фиксируются только те развилки, которые
документы не решают напрямую.

---

## Фаза 0

- **DB-адаптер: try/catch require('better-sqlite3'), fallback node:sqlite** —
  ТЗ §20.8 требует именно эту пару без деталей выбора момента проверки. Выбрал
  ленивую загрузку в `db.js:pickDriver()` (проверяется при первом `openDb()`,
  не при импорте модуля) — чтобы одна недоступная сборка не валила импорт
  `journal.js` целиком. better-sqlite3 на этой машине собрался нормально
  (Windows, Node 22.23.1) — оба драйвера проверены `evals/race.js`.
- **journal.db: WAL + busy_timeout=5000 на обоих драйверах** — ТЗ требует
  BEGIN IMMEDIATE для atomic capture; WAL снижает вероятность SQLITE_BUSY при
  параллельных воркерах (Фаза 6), busy_timeout — подстраховка вместо ручного
  retry-цикла на уровне журнала.
- **root_spec: ключ project_id (default 'default'), а не глобальный singleton**
  — ТЗ §4 описывает root_spec как факт, не даёт схему. Сделал таблицу с PK
  project_id, чтобы не блокировать многопроектность в будущем без миграции,
  при этом v1 всегда использует 'default'. `journal.js: setRootSpec/getRootSpec`.
- **skills.js: hallucinationGuard — regex по чужим расширениям + проверка,
  что файловые токены урока входят в touches_files задачи** — §4 описывает
  правило словами ("ссылается только на реальные файлы задачи"), конкретный
  алгоритм не дан. Реализовал как чистую функцию `skills.js:hallucinationGuard`,
  используется из `agents/skill.js` (Фаза 3) и из `writeSkill()` напрямую.
- **io.js: sanitizeLine здесь, а не в talk.js** — шрам 32 про санитизацию
  ввода сеанса привязан к слою чтения строк, а не к конкретной точке входа;
  так переиспользуется всеми будущими REPL (talk.js, resume.js).
- **mock.js в Фазе 0 — только каркас (MOCK_HANDLERS пуст, registerMockHandler
  экспортирован)** — агентов ещё нет, реальные ветки появляются вместе с
  ролями (Фаза 1: coder/tester; Фаза 3: advisor/router/architect/skill;
  Фаза 5.5: live_in), как явно указано в DEV_GUIDE §2.
- **package.json: better-sqlite3 в optionalDependencies, не dependencies** —
  чтобы `npm install` не падал целиком на машинах без toolchain для нативной
  сборки; адаптер и так рассчитан на отсутствие пакета.
- Два маленьких реальных вызова `claude -p --model haiku ...` (~$0.004
  суммарно) сделаны ДО фазы 1 для проверки недокументированных флагов CLI
  (`--system-prompt-file`, полный набор изоляции) — эти флаги не описаны в
  `claude -p --help`, только упомянуты в тексте `--bare`. Без проверки нельзя
  было доверять контракту шлюза. Дальнейшие реальные прогоны отложены в
  RUNBOOK.md по вашей просьбе.

## Фаза 1

- **Windows: `spawn('claude', args, {shell:true})` + ручное
  `JSON.stringify()`-квотирование каждого аргумента** — обнаружен реальный баг
  при сборке шлюза: прямой `spawn('claude', ...)` без shell даёт `ENOENT`
  (Windows не резолвит `.cmd`-шим по PATH без шелла), `spawn('claude.cmd', ...)`
  без shell даёт `EINVAL` (известная особенность Node на Windows для
  .bat/.cmd), а `shell:true` с аргументами-путями, содержащими пробел, обрезает
  аргумент по первому пробелу без ручного квотирования. Решение проверено
  вживую (см. `gateway.js:buildSpawnArgs`). На POSIX — обычный `spawn` без
  shell, без кавычек (кавычки на POSIX испортили бы аргумент, там шелл не
  участвует). Это и есть требуемая инвариантом §20.6 «Windows-ветка для
  claude.cmd», просто конкретный механизм не был описан в документах.
- **checkRunner: shell-обвязка критериев — всегда `bash -c`, включая Windows**
  — `bash` из Git for Windows резолвится напрямую (это настоящий `.exe`, не
  `.cmd`-шим), поэтому кросс-платформенно работает без shell:true и без
  Windows-квотирования. Соответствует прямому указанию DEV_GUIDE §0
  («bash-зависимости checkRunner документированы: Windows = Git Bash»).
- **workspace/package.json создаётся координатором автоматически, БЕЗ поля
  "type"** — реальный прогон MOCK-сценария калькулятора упал: Node ищет
  ближайший package.json вверх по дереву, находил корневой package.json LOOM
  (`"type":"module"`) и трактовал `calc.js` продукта как ESM, из-за чего
  `module.exports` кидал `ReferenceError`. Ни ТЗ, ни DEV_GUIDE это не
  упоминают — фикс в `coordinator.js:ensureWorkspacePackageJson`. LOOM не
  должен навязывать продукту клиента свою модульную систему.
- **Auto-commit safety net в варианте А — безусловный git-коммит харнессом
  после каждого хода кодера, а не «страховка на случай, если кодер забыл
  git_commit»** — в JSON-варианте А у кодера физически нет git-инструмента
  (нет tool_use), поэтому «забыть закоммитить» структурно невозможно; сам
  факт того, что коммитит харнесс, а не модель — и есть реализация принципа
  шрама 4 в этом варианте (§7: полноценный `submit_result`/git_commit —
  спящая спецификация варианта Б). `coordinator.js:autoCommit`.
- **Критерий = `{cmd, expect}` (expect опционален); JS-инлайн распознаётся
  строго по форме `node -e "<код>"` / `node --eval '<код>'` целиком** —
  DEV_GUIDE даёт контракт `{cmd, expect}` без грамматики извлечения. Регэксп
  матчит ТОЛЬКО команду, целиком состоящую из `node -e "..."`; остальные
  команды (`node --check file.js`, готовый `.cjs`, `curl`) идут в bash как
  есть — извлекать нечего, шрам 7 неприменим. `checkRunner.js:NODE_EVAL_RE`.
- **cli.js принимает свободный текст, `--file task.json` или флаги
  `--title/--spec/--criteria/--role/--type/--budget`** — DEV_GUIDE говорит
  только «задача задаётся руками в коде/аргументом», не давая формат. Выбрал
  три пути ввода: быстрый смоук (позиционный текст, без критерия — уйдёт в
  blocked_needs_human, что честно), файловый (для содержательных ручных
  прогонов вроде DoD sum.js — см. RUNBOOK.md), и флаговый.
- **mock.js: различение задач сценария калькулятора по содержимому
  промпта (`/calc\.js/`, `/index\.html/`, наличие секции «Отчёт тестера»),
  а не по глобальному счётчику вызовов** — счётчик вызовов ролей общий на
  процесс и не привязан к конкретной задаче; текстовое распознавание
  устойчиво к порядку обработки очереди координатором.
- **gateway.js: cost_usd берётся из `total_cost_usd` в JSON-конверте
  claude-cli напрямую** (подтверждено реальным вызовом) — резервная формула
  по `TIER_PRICES_USD_PER_1M` используется только если поле вдруг отсутствует.
  Это точнее и проще условной таблицы цен, о которой пишет §13 ТЗ.

## Фаза 2

- **«Ход» для правила дельты (§6.1) = номер попытки (task.attempts) ИМЕННО
  этой задачи**, не глобальный счётчик вызовов кодера. Ход 1 и ход 2 —
  полный снимок, ход 3+ — added/changed/removed относительно снимка,
  показанного на предыдущем ходу ЭТОЙ ЖЕ задачи. Состояние — in-memory
  `Map(task_id → snapshot)` в `agents/coder.js`; при рестарте процесса между
  попытками откатываемся к полному снимку — деградация в сторону БОЛЬШИХ,
  но корректных данных, не баг. Очищается по завершении задачи
  (`clearEyesState`) координатором, чтобы не течь памятью на длинных прогонах.
- **claimNext() больше НЕ инкрементирует attempts; инкремент — отдельная
  `incrementAttempts()`, вызываемая координатором ПОСЛЕ проверки бюджета,
  ПЕРЕД вызовом кодера.** Обнаружено при живой прогонке evals/budget.js:
  изначально attempts инкрементился прямо в захвате (Фаза 1), из-за чего
  задача, остановленная бюджетом ДО первого реального вызова кодера,
  показывала attempts=2 вместо 1 — захват без реальной попытки тратить деньги
  не должен считаться попыткой. Это меняет семантику `claimNext()` из Фазы 1
  (правка сделана в её же файле, не поверх — гонка по-прежнему защищена той
  же BEGIN IMMEDIATE транзакцией, `evals/race.js` зелёный после правки).
- **evals/budget.js: MOCK_CALL_COST_USD = 0.01 (config.js)** — в MOCK-режиме
  каждый вызов роли раньше логировался с cost_usd=0, что делает бюджет-стоп
  непроверяемым без реальных денег. Синтетическая стоимость 0.01 — тот самый
  «игрушечный лимит» из DoD DEV_GUIDE §4, позволяет проверить остановку
  детерминированно и бесплатно.
- **mock.js: coder-хендлер для задач вне сценария калькулятора возвращает
  файл-заглушку вместо throw** — чтобы evals/budget.js (и будущие MOCK-эвалы)
  могли переиспользовать общий MOCK_HANDLERS.coder, не расширяя сам
  сценарий-эталон калькулятора.
- **Проверка prompt-кэша и дельты tokens_in по avg() — отложены в RUNBOOK.md**
  как реальные прогоны (нужны настоящие вызовы claude-cli, MOCK всегда даёт
  tokens_in=0). Механизм логирования (cache_read/cache_creation в
  events.payload) уже на месте с Фазы 1 — RUNBOOK описывает точную команду
  проверки и SQL.

## Фаза 3

- **`\b` в JavaScript-регэкспах не распознаёт кириллицу как `\w`** — реальный
  баг, найденный дважды за эту фазу: (1) в MOCK-роутере классификация tweak
  по `/^поменяй\b/i` не матчилась на «поменяй вывод...» (граница
  буква→пробел не считается word boundary, когда буква кириллическая);
  (2) в `agents/advisor.js:TECH_QUESTION_RE` — «px/мс» не ловилось тем же
  образом, из-за чего код-уровневый guard шрама 15 фактически не работал бы
  на реальных технических вопросах с кириллицей рядом. Фикс — везде, где
  термин может стоять рядом с кириллицей, использовать
  `(?<![\p{L}\d])term(?![\p{L}\d])` с флагом `u` вместо `\b`. Это НЕ описано
  ни в ТЗ, ни в DEV_GUIDE — чисто платформенная особенность JS-регэкспов,
  но напрямую бьёт по шраму 15 (советник спрашивал px), поэтому фиксирую
  здесь как критичный для доверия к guard'у баг.
- **route на ошибку парсинга роутера — буквально `'project'`, хотя это не
  входит в перечень 5 типов входа §1** — DEV_GUIDE §1 прямо пишет
  `route=project`, приоритет у DEV_GUIDE при расхождении с ТЗ. В
  `agents/router.js` 'project' — синоним «безопасный полный цикл»:
  `talk.js` направляет его в тот же `runProjectFlow` с протоколом `wish`
  (запасной, самый нейтральный протокол интервью из трёх).
- **attempts инкрементируется и для type=regression** (в отличие от
  обычных задач, где attempts = число реальных вызовов кодера) — регрессия
  вообще не вызывает кодера (§8.9), но переиспользование того же счётчика
  не создаёт путаницы (регрессия почти всегда решается за 1 «попытку» —
  один прогон tester'а) и избавляет от отдельной ветки в handleAttemptFailure.
- **Регрессионная задача зависит (blocked_by_task_id) только от ПОСЛЕДНЕЙ
  обычной задачи дерева, не от всех** — схема журнала (§4 ТЗ, дословно)
  даёт только ОДНУ колонку зависимости, полноценный fan-in (несколько
  родителей) — явно Фаза 6. Доказано (и проверено `evals/race.js`-подобной
  логикой claimNext), что при строго однопоточной обработке claimNext
  ВСЕГДА берёт самую старую pending-задачу — значит к моменту, когда
  последняя обычная задача дойдёт до done, все более ранние уже в
  терминальном статусе. Для однопоточного Фаза 3 этого достаточно и корректно;
  бросается в глаза как упрощение, которое Фаза 6 обязана заменить настоящим
  fan-in (§16).
- **advisor.js: интервью в MOCK-режиме сразу возвращает `ready`** (кроме
  специального маркера `ТЕСТ_ТЕХ_ВОПРОС` для `evals/interview.js`) — чтобы
  `npm run mock` оставался быстрым и детерминированным; сам механизм
  многоходового интервью (`continueIntake`, предохранитель
  `MAX_INTERVIEW_TURNS`) проверен структурно (используется в `talk.js`),
  но многоходовый диалог с реальными уточняющими вопросами — предмет
  реального прогона (RUNBOOK).
- **mock.js: сопоставление задач сценария — по title из "## Задача", не по
  всему тексту промпта** — обнаружен реальный баг: секция критерия
  (`## Критерий приёмки`) часто содержит имена файлов вроде `calc.js`
  (потому что сама проверка их читает), из-за чего матчинг по всему
  `userMsg` ложно триггерился. Матчинг сузили до строки title.
- **skill_writer вызывается только на статус `done`, не на
  `blocked_needs_human`** — §4 говорит «после каждой завершённой задачи»;
  трактую «завершённая» как «done», а не любой терминальный статус — урок
  из проваленной задачи менее надёжен (причина провала могла быть внешней:
  бюджет, вопрос человеку), и §4 явно называет момент «после done» через
  описание координатора.

## Фаза 4

- **Docker — опциональный слой, включается автодетектом, а не жёсткая
  зависимость.** Демон Docker недоступен на этой машине (`docker info`
  падает, хотя CLI установлен) — по прямому указанию задания: «если docker
  недоступен в среде — реализуй код полностью, проверку оставь в
  RUNBOOK.md». `checkRunner.runCheck()` по умолчанию использует
  `sandbox:'local'` (безопасный, уже проверенный путь); `coordinator.js`
  сам решает через `ensureDockerSandbox()` — собирает образ РОВНО когда
  демон отвечает И сборка успешна, иначе явно передаёт `sandbox:'local'`.
  Ничего не «auto-детектится» неявно внутри checkRunner — иначе состояние
  «демон есть, а образ не собран» тихо ломало бы тестер (docker run без
  образа полез бы тянуть несуществующий тег из registry).
- **Образ workspace пересобирается один раз на вызов `runCoordinatorLoop`
  для данного workspaceDir, не на каждую задачу** — сборка кэшируется
  Docker по слоям (быстро, если package.json не менялся), но правило
  «пакет понадобился посреди дерева → статус «нужен пакет» → пересборка»
  (§12.2) в этой фазе реализовано КАК РУЧНОЙ путь (новый `/resume` заново
  входит в `runCoordinatorLoop` и пересобирает), не как автотриггер на
  каждое изменение `package.json`. Полная реактивность — по потребности,
  не специфицирована жёстко ни в ТЗ, ни в DEV_GUIDE для Фазы 4.
- **architect.js по-прежнему делает `npm install` напрямую на хосте (execFileSync)**
  вдобавок к тому, что тот же `package.json` попадает в Docker-сборку —
  сознательное дублирование: хостовый install остаётся ЕДИНСТВЕННЫМ
  работающим путём, когда Docker недоступен (наш подтверждённый тестами
  случай), а Docker-сборка ставит те же зависимости ВТОРОЙ раз изолированно
  внутри образа. Не конфликтует (разные файловые системы), не убираю
  хостовый install, чтобы не сломать единственный сейчас проверяемый путь.
- **gitGuard проверяет ветку РЕПОЗИТОРИЯ LOOM (ROOT), не workspace** — шрам 5
  буквально про репозиторий LOOM («репо продукта» — отдельная история,
  Фаза 5/GitHub). `coordinator.gitGuard()` — pre-flight в начале
  `runCoordinatorLoop()`, бросает исключение (не тихо продолжает), если
  ветка LOOM не main/master; если ROOT вообще не git-репозиторий — не
  блокирует (нечего защищать, напр. тесты во временных каталогах).
- **tsconfig.json — заготовка, НЕ подключена к координатору** (DEV_GUIDE
  §6: «включается в Фазе 6»). Присутствие файла — чтобы Фаза 6 не начинала
  с нуля; `tsc --noEmit` level-0 нигде не вызывается автоматически сейчас.
- **Реальная проверка `rm -rf /` внутри контейнера не трогает хост —
  отложена в RUNBOOK.md** (нужен работающий демон Docker, недоступен в
  этой сессии). Путь замка (`../evil.js` блокируется и логируется событием
  `path_lock_blocked`) проверен и зелён в MOCK — это чисто харнесс-уровневая
  защита, Docker не нужен для её работы (действует уже с Фазы 1, §12.1
  «до Docker — временная решётка»).

## Фаза 5.5

- **live_in: Playwright — опциональная зависимость (`optionalDependencies`,
  как better-sqlite3), с фолбэком на статический разбор HTML.** В этой
  среде `npm install playwright` не выполнялся (браузерные бинарники
  тяжёлые, не нужны для проверки MOCK-веткой) — `agents/livein.js:
  inspectWithPlaywright()` пробует `import('playwright')`, при неудаче
  честно откатывается на `inspectStatic()` (читает HTML напрямую,
  помечает результат `live:false` и текстовым `note`, чтобы промпт
  live_in знал о сниженной надёжности данных). Живой Playwright-прогон —
  RUNBOOK-пункт (после `npm install && npx playwright install chromium`).
- **live_in создаёt задачи `type='polish'`, не `'tweak'`** — §3 (таблица
  ролей) говорит «создаёт tweak-задачи», а §9 говорит «tweak/polish». Схема
  журнала (§4) явно перечисляет `polish` как отдельный тип — выбрал его,
  т.к. он точнее отличает «сам себя нашёл на обживании» от «клиент попросил

### П§2 (миграции журнала)

- **`ensureColumn(d, table, column, ddl)` — PRAGMA table_info вместо
  try/catch на ALTER TABLE** — SQLite не даёт `ADD COLUMN IF NOT EXISTS`;
  try/catch проглотил бы и настоящие ошибки (опечатка в DDL, блокировка
  файла), не только «колонка уже есть». Явная проверка списка колонок —
  тот же паттерн, что уже применён к таблицам (`CREATE TABLE IF NOT
  EXISTS`), просто для ALTER TABLE эквивалента нет. `journal.js:migrate()`.
- **DoD «зелёный на обоих драйверах» проверен только фактическим
  better-sqlite3 на этой машине** — переключателя «форсировать
  node:sqlite» в кодовой базе нет (`db.js:pickDriver()` — try/catch на
  require, а не флаг), добавлять его ради одной проверки миграции — за
  рамками П§2. Вся новая DDL (CREATE TABLE IF NOT EXISTS, ALTER TABLE ADD
  COLUMN, PRAGMA table_info) — стандартные SQLite-операции без
  драйверно-специфичного синтаксиса, тем же классом, что уже проверенные
  на обоих драйверах таблицы Фазы 0 (DECISIONS Фаза 0: «оба драйвера
  проверены evals/race.js»). Реальная проверка node:sqlite конкретно для
  tools/task_deps — RUNBOOK, если/когда better-sqlite3 не соберётся на
  какой-то машине естественным образом.
- **Идемпотентность и отсутствие потери данных проверены на КОПИИ реальной
  `journal.db` этой сессии** (23 задачи, 162+ событий на момент проверки):
  скопировал файл, применил новую `migrate()`-логику дважды подряд (второй

  на промпт.
- **События `tool_run`/`tool_registered`/`tool_network` — литеральные
  значения колонки `events.type`**, не вложены в `payload.event` под общим
  `type='status'`/`'action'` (как уже сделано для `claimed`/
  `path_lock_blocked`/`max_attempts_reached` в `coordinator.js` с Фазы 3).
  ТЗ v4 §4 перечисляет `thinking|action|test_result|status|usage` как
  ПРИМЕР, не CHECK-constraint (в схеме её нет — проверено). DoD П§4 прямо
  требует «события tool_registered/tool_run в events», читаемое как
  `SELECT * FROM events WHERE type IN (...)` без парсинга JSON — выбрал
  буквальное прочтение DoD.
- **Архитектор создаёт И `type:'tool'`, И `type:'tool_run'` задачи**, хотя
  §26.2 п.1 буквально говорит только про `type:'tool'` для архитектора
  (`tool_run` официально — путь researcher'а, §26.2 п.2). MOCK-сценарий
  26.6 прямо требует «архитектор выдаёт DAG с задачей type:'tool' + задачей
  type:'tool_run' + обычной задачей» — без этого инструмент, собранный
  архитектором внутри одного дерева, некому было бы вызвать в том же
  дереве. `prompts/architect.md` объясняет разницу и требует
  `depends_on_title`-цепочку от tool → tool_run → потребитель.
- **MOCK-фикстура demo.txt создаётся сценарием напрямую (`fs.writeFileSync`
  в `runForgeScenario`), не отдельной задачей дерева** — избегает
  необходимости в множественной зависимости (tool_run зависел бы И от
  сборки инструмента, И от задачи, создающей demo.txt — task_deps это уже
  умеет с П§2, но городить это только ради демонстрации Кузницы —
  избыточно). Реальный сетевой/многозвенный кейс — предмет Фазы 7
  (researcher), не этой фазы.
- **Docker-песочница для `runTool()` НЕ реализована в этой фазе** — §26.3
  п.4 сама формулирует это как честное ограничение («локальный путь сеть
  физически не enforce'ит»), тот же паттерн, что уже принят для
  `checkRunner` в DECISIONS Фазы 4 (Docker — опциональный слой, демон
  недоступен в среде сборки). `tool_network`-событие логируется как факт
  аудита уже сейчас; сам Docker-путь для tools — по потребности, когда
  демон появится в среде (тот же RUNBOOK-пункт, что и для checkRunner).

### П§5 (Фаза 6 — параллельность, §16)

- **Реальный баг №1, найден MOCK-сценарием: `git merge --no-ff` в ветку БЕЗ
  единого коммита падает** («Non-fast-forward commit does not make sense
  into an empty head») — не конфликт, техническая ошибка. Однопоточный путь
  этого никогда не задевал (там нет `merge --no-ff`, только `autoCommit`);
  параллельная фаза создаёт worktrees и мёржит В main ДО того, как кодер
  вообще успел что-то закоммитить. Фикс: `coordinator.js:ensureInitialCommit()`
  — явный стартовый коммит перед первым `ensureWorktree()`, вызывается
  только из параллельной фазы (однопоточный путь не тронут).
- **Реальный баг №2, найден ТЕМ ЖЕ MOCK-прогоном сразу после фикса №1:
  `ensureInitialCommit` с `--allow-empty` БЕЗ `git add` оставлял
  `package.json` (уже написанный `ensureWorkspacePackageJson` НА ДИСК main
  до этого коммита, Фаза 1) неотслеженным.** Worktrees ветвились от
  коммита БЕЗ package.json в истории; каждый воркер, работая в своём
  worktree, находил, что package.json «отсутствует», и создавал СВОЙ
  (та же `ensureWorkspacePackageJson`, вызванная уже для worktree-пути) —
  при мёрже git отказывался переписать untracked-файл main'а версией из
  ветки («would be overwritten by merge», данные могли бы потеряться).
  Фикс: `git add -A` перед стартовым коммитом — всё, что уже лежит на
  диске main на момент первого входа в параллельную фазу, становится
  частью истории, worktrees наследуют его сразу. Оба бага — ровно то, для
  чего нужен MOCK ПЕРЕД реальным прогоном (§24): нашлись за секунды,
  бесплатно, до единого реального вызова модели.
- **`claimed_by` теперь НЕ обнуляется при переходе в `merge_pending`**
  (только раньше обнулялся при ЛЮБОМ статусе, кроме `claimed`) —
  `core/merge.js:mergeOneTask` читает `task.claimed_by`, чтобы понять,
  КАКОЙ worktree/branch (`loom/w<i>`) сливать; без фикса это поле уже было
  бы `NULL` к моменту, когда `drainMergeQueue` добирается до задачи (баг
  найден при проектировании, до первого прогона — `journal.js:setStatus`).
- **regression/tool_run исключены из параллельного захвата** через новый
  параметр `claimNext(..., {excludeTypes})`, не через отдельный «peek и
  вернуть в pending» — чище (не тратит фиктивный захват) и симметрично
  `activeTouches` (тот же объект опций). Обе роли структурно нуждаются в
  `workspaceDir=main` (`tool_run` реально исполняет инструмент с этим cwd;
  regression проверяет ИНТЕГРИРОВАННОЕ состояние продукта) — в worktree
  воркера они увидели бы чужую, не залитую в main версию. §16 п.8 говорит
  это же про регрессию текстом («строго после пустой очереди мёржей»);
  `tool_run` там прямо не упомянут, но по конструкции (§26.3 п.1: cwd =
  workspace текущего продукта) требование то же самое.
- **Искусственный конфликт в MOCK-сценарии — через НЕДОдекларированный
  touches_files, не через одинаковый touches_files у обеих задач.** Если
  бы обе конфликтующие задачи честно объявили `shared.js` в touches_files,
  weave вообще не дал бы им попасть в один раунд — сценарий тестировал бы
  только вход в weave-фильтр, не реальный git-конфликт и reconciler. Это
  одновременно и работающий тест, и честная демонстрация ГРАНИЦЫ file-level
  weave (§16 п.3 прямо признаёт это ограничение — touches_functions/
  tree-sitter сознательно за рамками v1.0-final): конфликты, невидимые
  декларации, всё ещё возможны, и именно поэтому reconciler — не опция, а
  обязательный второй рубеж защиты.
- **tsc-ворота: `resolveTscBin()` ищет ТОЛЬКО локально установленный `tsc`
  (`node_modules/.bin/`), никогда не вызывает `npx tsc` напрямую** — `npx`
  без предустановленного пакета молча качает его из сети, что прямо
  запрещено в задании для MOCK/без RUN_REAL. `typescript` добавлен в
  `optionalDependencies` (тот же паттерн, что `better-sqlite3`/
  `playwright`) — не установлен в этой среде сборки, ворота корректно
  graceful-degrade (`{ok:true, skipped:true}`), не блокируют merge
  недоступностью инструмента. MOCK-проверка «красной» ветки ворот — через
  `evals/fixtures/fake-tsc-impl.js`, подставленный как `node_modules/.bin/tsc(.cmd)`
  в тестовую директорию: реально исполняется как дочерний процесс (не
  замокан вызов функции), просто не настоящий `tsc` — тот же принцип, что
  `runTool` в §26.6 («реальный локальный процесс без модели и сети —
  мокать нечестно и не нужно»), применённый туда, где САМ бинарь
  недоступен по независящей от логики причине.
- **`depends_on_titles` (массив) — новое поле архитектора, ДОПОЛНЯЕТ
  `depends_on_title` (одиночный), не заменяет** — П§5 требует «touches_files
  обязателен, deps-массив» без уточнения имени поля/формата. Плюрализация
  по аналогии с уже существующим полем (не ломает старые DAG без этого
  поля) — `agents/architect.js` мапит оба на `deps` (task_deps, П§2) при
  создании задачи; `claimNext` уже требует одновременного выполнения ОБЕИХ
  форм зависимостей (П§2), новое поле просто даёт архитектору способ
  ЗАПОЛНИТЬ вторую форму для НЕ-регрессионных задач (раньше это делал
  только harness для авто-регрессии).
- **`evals/race.js`: внутрипроцессная weave-гонка НЕ доказывает атомарность
  BEGIN IMMEDIATE заново** (это уже честно доказано кросс-процессным
  сценарием — реальные ОС-процессы, реальная конкуренция за файл БД) —
  она проверяет, что JS-фильтр по `activeTouches` корректен именно в
  паттерне вызова, каким его использует `coordinator.js:claimAndRunInWorktree`
  (несколько `claimNext` из одного `Promise.all`-раунда, каждый со свежим
  `activeTouches`). 2 файла давали только 2 захвата за раунд независимо от
  числа воркеров (не баг — честное свойство file-level weave); тест
  переписан на 4 файла = 4 воркерам, чтобы раунды типично утилизировали
  всех воркеров, и на цикл до исчерпания очереди вместо фиксированного
  числа раундов (предохранитель — не бесконечный, явный FAIL при поломке).

### П§6 (Фаза 7 — глубина входа: researcher, ingest, банк/judge)

- **Весь стейт researcher-цикла — на доске (task/events), не в памяти
  процесса** (§10, п.6.1): `need_tools`/`tool_runs` из звонка 1 материализуются
  как обычные задачи `type='tool'`/`type='tool_run'` ДО их исполнения, звонок
  2 читает `task.feedback` уже done-задач + read-only глаза продукта — тот же
  принцип, что и у остального харнесса (журнал — источник правды, §4),
  просто применённый к новой роли.
- **`FICTIONAL_TOOL_CRITERIA`/`FICTIONAL_DATA_CRITERIA` (agents/researcher.js)
  — честный FAIL-заглушка, если researcher не прислал `criteria`/
  `data_criteria`.** Не выдумываем правдоподобный критерий за модель (это
  было бы симулированной проверкой, правило 5 §8) — задача просто гарантированно
  не сможет считаться выполненной, пока кто-то не поправит промпт/модель;
  предпочли явный, диагностируемый отказ тихому прохождению.
- **Стоп-фактор `data_unavailable` — структурно единственный путь при
  недоступности данных**: `bin/talk.js:runProjectFlow` / `bin/ingest.js`
  не содержат ветки «продолжить без данных» вообще — `return` до вызова
  архитектора. Так §10 инвариант выражен кодом, а не только промптом.
- **MOCK: маркер OK/NODATA для researcher-оценки прокидывается через
  `tool_runs[0].args` (видны в title задачи), не через текст брифа.**
  Причина: спека прямо говорит, что звонок 2 (оценка) видит ТОЛЬКО отчёты
  tool_run + глаза продукта, НЕ исходный текст problem — маркер в брифе был
  бы недоступен на этом звонке физически, а протаскивать его отдельным
  скрытым каналом мимо контракта — нечестный мок. Через `args`→`title` —
  честно, тем же путём, каким узнал бы и человек, читающий отчёт задачи.
- **`covers:[reqId]` — новая колонка `tasks.covers` (не отдельная таблица)**:
  тот же паттерн, что `tasks.tool_name` (П§2 `ensureColumn`, идемпотентно на
  существующей БД). Требование без единой покрывающей задачи ищется в JS
  (`agents/architect.js`: множество всех `covers` минус список id
  требований) — не блокирует прогон, только отчётность человеку
  (`uncoveredRequirements` в результате `runArchitect`), симметрично тому,
  как judge (`evals/judge.js`) — метрика, не ворота.
- **`bin/ingest.js` — самостоятельная точка входа, НЕ импортирует
  `bin/talk.js`.** `talk.js` вызывает `main()` безусловно на верхнем уровне
  модуля (запускает REPL) — импорт запустил бы чужой REPL как побочный
  эффект импорта. Общий хвост (architect→coordinator→live_in→отчёт→publish)
  поэтому вынесен внутри `talk.js` в `runBuildFlow()` (используется и
  `/ingest`, и обычным project-потоком), а `bin/ingest.js` — отдельный
  скрипт с тем же хвостом, продублированным по образцу уже существующего
  `bin/cli.js` (который тоже не переиспользует код `talk.js` из-за того же
  ограничения) — альтернатива (экспортировать `runBuildFlow` и обвязку из
  `talk.js`, обернув REPL-старт в `if (import.meta.url===...)`) была
  отклонена как более рискованная правка стабильно работающего файла ради
  вторичного CLI-входа.
- **MOCK-разбор документа (ingest) читает универсальный формат `- текст`
  построчно, а не подстраивается под конкретную фикстуру.** Мок-советник в
  режиме «разбор документа» распознаёт requirement-строки по markdown-списку
  (`^- `), что работает для ЛЮБОГО документа в этом формате, не только для
  `evals/fixtures/spec-package/`; это ближе к реальному поведению модели
  (которая тоже будет ориентироваться на структуру списка), чем матчинг по
  содержимому конкретных файлов.
- **Фикстура `evals/fixtures/spec-package/` содержит ДВЕ независимые дыры,
  проверяющие ДВА разных механизма:** `TODO_HOLE`-маркер в
  `02-payment.md` — требование без проверяемого критерия, ловится ingest'ом
  ДО архитектора (§11 п.3); мок-архитектор нарочно не покрывает ПОСЛЕДНЕЕ
  по счёту требование (`03-design.txt:R1`) — ловится трассировкой `covers`
  ПОСЛЕ архитектора (§11 п.4). Разные req_id намеренно — тест доказывает,
  что harness различает «дыра в самом ТЗ» и «архитектор забыл про
  требование», а не просто повторно детектирует одну и ту же дыру дважды.
- **`evals/bank.js`: под MOCK route НЕ проверяется, только структура
  интервью.** Мок-роутер — детерминированный регэксп (не настоящая
  диагностика), умеющий отличить problem от wish только по спец-маркеру
  (`ТЕСТ_ИССЛЕДОВАТЕЛЬ`, уже используется researcher-сценарием
  `core/mock.js`); банк намеренно использует ЕСТЕСТВЕННЫЙ текст желаний
  (не маркеры) — под MOCK протокол берётся из фикстуры напрямую
  (`protocol_expected`), `route_expected` сверяется только при `MOCK=false`
  (полный релизный прогон с живой моделью, см. RUNBOOK «выпускной экзамен»).
  Тот же принцип уже применяет `evals/interview.js`.
- **`evals/judge.js` логирует событие `eval_score` ДАЖЕ при сбое шлюза**
  (`result:null, error:"..."`) — отсутствие оценки тоже сигнал (модель
  недоступна/лимит), выбрасывать его молча означало бы терять данные для
  последующего анализа промптов (§14.3).
- **pdf/бинарные макеты в ingest — честная строка-заглушка
  (`agents/ingest.js:readDocuments`), реальный парсинг pdf — сознательно
  за рамками v1.0-final** (дословно оговорено в самом DEV_GUIDE part2 §6.2
  п.1 — фиксируется здесь для единообразия формата решений).
