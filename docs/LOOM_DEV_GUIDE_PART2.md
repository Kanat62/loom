## П§1 = §26 ТЗ. Кузница инструментов (Tool Forge)

> Статус раздела: полноправная часть ТЗ v4. При расхождениях с частью 1
> DEV_GUIDE приоритет у этого раздела (он новее и уже).

### 26.0. Зачем и что это

LOOM не имеет рук, кроме тех, что построил сам. Кузница — механизм, которым
система создаёт себе недостающие способности: парсер площадки, конвертер
данных, генератор скриншотов, проверятор фида — что угодно, что выражается
исполняемым скриптом. Философия прежняя (§2): **вся агентность — в
детерминированном коде**. Модель никогда не исполняет инструмент сама — она
только (а) *заказывает* инструмент задачей на доске и (б) *получает результат*
его исполнения в контекст следующего хода. Исполняет — харнесс, в песочнице.

Инструмент = продукт, у которого клиент — сам LOOM. Поэтому к нему применяется
ВЕСЬ существующий конвейер без исключений: пишет кодер (вариант А), критерии
пишет архитектор по Своду §8, проверяет тестер, симуляции запрещены (§8.5),
бюджет и лестница попыток — как у любой задачи. Никакого «особого доверенного
пути» для инструментов не существует — доверие построено на недоверии.

### 26.1. Реестр

Файловая часть: `tools/<name>/` в корне LOOM (НЕ в workspace — инструменты
переживают проекты). Внутри минимум два файла:

- `run.js` — точка входа. **Только stdlib Node 22** (включая глобальный
  `fetch`); npm-зависимости инструментам запрещены в v1 (иначе каждый
  инструмент тащит свой node_modules и ломает изоляцию — расширение через
  общий tools/package.json допустимо позже отдельным решением).
  Контракт исполнения: аргументы — позиционные строки; результат — stdout
  (данные пишутся в файл, путь к которому передан аргументом) и код выхода
  (0 = успех). Никакого интерактива.
- `tool.json` — манифест:
  ```json
  {
    "name": "fetch-cars-listing",
    "version": 1,
    "description": "скачивает N объявлений площадки X в JSON с полями марка/год/цена",
    "args": ["outFile", "maxItems"],
    "network": "fetch",
    "timeout_ms": 60000
  }
  ```
  `network`: `"none"` (по умолчанию) | `"fetch"` (инструменту нужна сеть).

Табличная часть — таблица `tools` в journal.db (не отдельная БД: регистрация
транзакционно связана с задачами):

```sql
CREATE TABLE IF NOT EXISTS tools (
  name TEXT PRIMARY KEY,
  version INTEGER DEFAULT 1,
  description TEXT,
  entry TEXT DEFAULT 'run.js',
  network TEXT DEFAULT 'none',
  timeout_ms INTEGER DEFAULT 60000,
  created_by_task TEXT,
  status TEXT DEFAULT 'active',     -- active | deprecated
  usage_count INTEGER DEFAULT 0,
  last_used_at INTEGER,
  created_at INTEGER
);
```

Реестр — единственный источник правды о том, какие инструменты существуют.
Модель, сославшаяся на незарегистрированный инструмент, получает
детерминированную ошибку со списком существующих (лекарство от галлюцинаций
имён — тот же класс, что шрам 10).

### 26.2. Жизненный цикл: заказ → сборка → регистрация → использование

**Заказ.** Инструментные задачи создают только два пути, оба — харнессом:
1. Архитектор включает в DAG задачу `type:'tool'` с полем `tool_name`, когда
   для критерия или для данных нужна отсутствующая способность. В user-контекст
   архитектора всегда подаётся текущий реестр (имя + описание) с железным
   правилом **reuse-first**: существующий инструмент переиспользуется, а не
   дублируется.
2. Researcher (§10) возвращает `need_tools` — харнесс превращает их в те же
   задачи `type:'tool'` + связанные `type:'tool_run'`.

Кодер в варианте А задач создавать не может структурно — поэтому **глубина
рекурсии Кузницы = 1 обеспечена архитектурой**, а не проверкой: задача-инструмент
не может породить другую задачу-инструмент. Это инвариант.

**Сборка.** `type='tool'`, `role='coder'`. Отличия от обычной задачи — только
в детерминированной обвязке координатора:
- целевой корень записи — `tools/<tool_name>/` вместо workspace; замок пути
  действует относительно этого корня (тот же `isPathInsideWorkspace`, другой
  base);
- критерий приёмки обязан РЕАЛЬНО исполнять инструмент (`node
  <toolsDir>/<name>/run.js out.json 50` + проверка результата) — §8.5 без
  скидок; для сетевых инструментов критерий-прогон и есть первая добыча данных
  (§10: «выдаёт ≥50 записей с полями марка/год/цена»);
- зелёный тестер → координатор вызывает `registerTool()` (upsert: имя занято и
  описание отличается → version+1, статус прежней записи не трогается — история
  в git tools-репозитория), пишет событие `tool_registered`.

**Использование.** `type='tool_run'`, `role='coder'`, но кодер НЕ вызывается —
та же ветка координатора «пропустить кодера», что уже существует для
`type='regression'`. `task.spec` = JSON `{"tool":"name","args":[...]}`;
координатор исполняет `runTool()`, кладёт усечённый stdout/stderr в
`task.feedback` (лимит 1000 — шрам 20), затем гоняет обычные `criteria` задачи —
для данных это критерии данных из §10 (свежесть / объём / правдоподобие).
Провал критериев = провал задачи, лестница попыток обычная (повторный запуск
инструмента — законная «попытка»: сеть могла мигнуть).

### 26.3. Исполнение: runTool

`core/tools.js`:

```
registerTool(manifest)                 → запись/апгрейд в tools
getTool(name) / listTools()            → чтение реестра
findTools(query)                       → LIKE по name+description (FTS5 — Фаза 8, вместе со skills)
runTool(name, args, {cwd, sandbox})    → { ok, code, stdout, stderr, duration_ms }
```

Правила `runTool` (инварианты Кузницы, нарушение = баг):
1. Инструмент исполняется ТОЛЬКО дочерним процессом
   (`node <ROOT>/tools/<name>/run.js …args`, cwd = workspace текущего проекта,
   чтобы данные ложились внутрь продукта). **Никогда** `require`/`import`
   кода инструмента в процесс координатора — инструмент не должен уметь
   дотянуться до journal.db/prompts даже теоретически (§12.6).
2. Окружение — минимальное (PATH и всё), `.env` LOOM внутрь не передаётся:
   секреты оркестратора инструментам не принадлежат.
3. Таймаут — из манифеста (потолок 300с, как MODEL_TIMEOUT); stdout режется
   до 4000 символов для контекста, полный вывод — только в файл, если
   инструмент сам его пишет.
4. Песочница — та же лестница, что у checkRunner: Docker при доступном демоне
   (workspace смонтирован rw, `tools/<name>` — read-only; `--network none`
   при `network:"none"`, дефолтная сеть при `"fetch"`), иначе локальный
   запуск. Честное ограничение фиксируется: локальный путь сетевую политику
   НЕ enforce'ит физически — только логирует событие `tool_network` (полный
   egress-allowlist — вместе с §12.2, когда Docker станет обязательным).
5. Каждый запуск — событие `tool_run` в events (name, duration, code) +
   `usage_count`/`last_used_at` в реестре. Диагностика инструментов = SQL,
   как всё остальное (§14).

### 26.4. Git и шрам 5

`tools/` добавляется в `.gitignore` репозитория LOOM, и внутри `tools/`
координатор делает собственный `git init` при первой регистрации. Итог:
агентский код физически не попадает в репозиторий машины (gitGuard спокоен,
шрам 5), но история инструментов версионируется отдельным репо —
`autoCommit` туда же, сообщением `loom-tool: <name> v<version> (<task.id>)`.

### 26.5. Стоп-факторы и этика

- Инструмент не собрался за MAX_ATTEMPTS → `blocked_needs_human`; зависимые
  tool_run и всё, что на них завязано, честно стоят. Выдумать данные вместо
  упавшего инструмента — худший провал системы (§10), симулировать «инструмент
  как бы отработал» — тот же грех.
- Сетевые инструменты: только публичные данные, User-Agent честный,
  robots/ToS уважаются, обходов защит нет (§10-этика). Шаблон сетевого
  инструмента в промпте кодера обязан содержать проверку robots.txt для
  корневого пути и паузу между запросами.
- Бюджет: задачи Кузницы живут в общем бюджете проекта (§13) без отдельного
  котла; дефолт задачи — стандартные 2.0.

### 26.6. MOCK-ветка Кузницы (§24 обязателен)

Сценарий в mock.js: архитектор выдаёт DAG с задачей `type:'tool'`
(`tool_name:'word-count'`, критерий реально запускает инструмент на эталонном
файле) + задачей `type:'tool_run'` + обычной задачей, чей критерий читает
результат tool_run. Мок-кодер возвращает детерминированный `run.js` (первая
попытка — с нарочной ошибкой: неверный код выхода; демонстрация цикла
поймал→отчёт→починил). **Сам runTool в MOCK исполняется по-настоящему** —
это детерминированный локальный процесс без модели и сети, мокать его нечестно
и не нужно. Проверяется: регистрация в tools, событие tool_run, reuse во
второй задаче, version-bump при повторном заказе с изменённым описанием.

## П§2. Миграции журнала (фундамент фаз 5.7 и 6)

Одна правка `journal.js: ensureSchema`, идемпотентная (через
`PRAGMA table_info` / `CREATE TABLE IF NOT EXISTS`), БЕЗ отдельного
migration-фреймворка:

1. `ALTER TABLE tasks ADD COLUMN tool_name TEXT` (26.2).
2. Таблица `tools` (26.1).
3. Таблица настоящего fan-in:
   ```sql
   CREATE TABLE IF NOT EXISTS task_deps (
     task_id TEXT NOT NULL,
     blocked_by TEXT NOT NULL,
     PRIMARY KEY (task_id, blocked_by)
   );
   ```
   `addTask` принимает `deps: [id,…]`; колонка `blocked_by_task_id` остаётся
   рабочей (обратная совместимость: одиночная зависимость может жить в ней),
   `claimNext` требует, чтобы И колонка, И все строки task_deps были в
   `done|merged`. Регрессионная задача с этого момента зависит от ВСЕХ задач
   дерева, а не только от последней (закрытие упрощения из DECISIONS Фазы 3).
4. Статус `merge_pending` уже перечислен в §4 — код должен начать его
   реально использовать в Фазе 6; до неё done-путь не меняется.

**DoD:** `npm run race` зелёный на обоих драйверах; старый journal.db
открывается новой схемой без потери данных (проверить на копии реальной базы
из первого живого прогона).

Тонкости, которые нельзя потерять:
- Секция в coder.md — СТАТИЧНАЯ (кэш, §6.3): сам факт «это задача-инструмент»
  и целевое имя приходят динамикой в user (текст задачи), но правила формата —

- Предпроверка критериев (§8.8) для tool-задач обязана выполняться с тем же
  cwd-контрактом, что боевой прогон (инструмента ещё нет → критерий обязан

- `clearEyesState` и глаза кодера для tool-задач строятся от tools/<name>/
  (обычно пусто на первой попытке) + read-only списка имён существующих
  инструментов — чтобы кодер видел соглашения соседей, но не их код целиком
  (экономия контекста, §6.3-принцип релевантности).

**Реальный прогон (обязателен, дешёвый):** через cli — задача-инструмент
`word-count` (вход: путь файла; выход: число слов в stdout; критерий: прогон
на эталонном файле из evals/fixtures → ожидаемое число) → done, запись в
tools, затем tool_run по нему. Сетевой инструмент в реальном прогоне —
отложить до Фазы 7 (там он нужен по делу).
**DoD:** mock зелёный со сценарием 26.6; `SELECT * FROM tools` показывает
регистрацию с created_by_task; события tool_registered/tool_run в events;

## П§5. Фаза 6 — параллельность (дни 7–12)

> Включать, когда однопоточный режим «скучно-стабилен» (Г§9) — т.е. после
> того, как реальные прогоны RUNBOOK части 1 пройдены и зелены.

Файлы: `core/worktrees.js` (новый), `core/merge.js` (новый: очередь +
reconciler), правки `core/coordinator.js` (N воркеров), `core/journal.js`
(weave-фильтр в claimNext), `agents/architect.js`/`prompts/architect.md`
(touches_files обязателен в DAG, deps-массив), `core/config.js`
(CODER_WORKERS, WORKTREES_DIR — тоже MOCK-изолированный).

Дизайн (в терминах существующего кода):

1. **Воркеры — в одном процессе.** `runCoordinatorLoop({workers:N})` →
   `Promise.all` N циклов `processOneTask`, каждый с
   `claimedBy='w'+i` и своим worktree. Отдельные процессы не нужны:
   BEGIN IMMEDIATE уже защищает захват (evals/race.js это доказывает для
   межпроцессного случая — внутрипроцессный строго слабее).
2. **Worktrees продукта.** `core/worktrees.js`: workspace — базовый git-репо;
   на воркера — `git worktree add .worktrees/w<i> -b loom/w<i>` от main.
   Кодер и тестер воркера живут в своём worktree (все существующие функции
   уже принимают workspaceDir явно — менять сигнатуры не надо).
3. **Weave по файлам.** `claimNext(role, claimedBy, {activeTouches})`:
   координатор передаёт объединение touches_files всех задач в
   `claimed|merge_pending`; SQL-кандидат дополнительно фильтруется в JS по
   пересечению массивов (JSON в SQLite парсить в SQL — хрупко). Пересечение →
   кандидат пропускается, берётся следующий. touches_functions/tree-sitter —
   сознательно НЕ в v1.0-final (нативная сборка tree-sitter — новый класс
   рисков ради выигрыша, который file-level weave уже даёт; зафиксировать в
   DECISIONS, дверь §16 остаётся открытой).
4. **mergeQueue — последовательный, в главном цикле.** Зелёный тестер в
   worktree → статус `merge_pending` (не done). Одна очередь: main-цикл
   берёт merge_pending по одному → `git -C workspace merge --no-ff loom/w<i>`
   → зелёно: статус `merged` (терминальный «done»-эквивалент: claimNext уже
   принимает оба), rebase остальных живых worktree на новый main.
5. **tsc level-0 (§15):** если в workspace есть tsconfig/jsconfig —
   `tsc --noEmit` как ворота ПЕРЕД merge; красный → merge не происходит,
   задача в failed с отчётом tsc (это и есть declares_interface).
6. **Конфликт:** авто-мёрж непересекающихся уже даёт git; конфликт →
   `core/merge.js: reconcile()` — один вызов `chat('cheap', …, {json:true})`
   с обеими версиями конфликтных файлов + текстами обеих задач; контракт
   ответа: `{"files": {"path": "ours"|"theirs"}}` (§16: ~94% решается выбором
   существующего варианта — генерация нового кода реконсайлеру ЗАПРЕЩЕНА,
   это не кодер); не решилось / ответ кривой → задача `blocked_needs_human`,
   worktree сохраняется для человека.
7. **Fan-in политика (§16):** перед каждым захватом координатор проверяет:
   есть failed, блокирующий (через task_deps) недостигнутые done-ветки →
   весь проект `blocked_needs_human`; иначе failed живёт в обычной лестнице.
8. **Регрессия и live_in — строго после пустой очереди мёржей** (один поток,
   по main): параллелить проверку целого — бессмыслица.

**MOCK:** сценарий «8 задач / 2 воркера»: mock-кодер детерминирован,
искусственный конфликт двух задач в одном файле (обе пишут разные строки в
shared.js) → mock-reconciler выбирает «theirs»; проверяется: ни одного
двойного захвата (по events), очередь мёржей строго последовательна
(timestamps merge-событий), tsc-ворота на подсунутом битом d.ts.
**DoD:** Г§9 дословно + `npm run race` расширен внутрипроцессной гонкой
(два claimNext из Promise.all).

### 6.1. Researcher на Кузнице

Роль в ROLES уже есть (`researcher: sonnet→opus`). Двухзвонковый цикл,
весь стейт — на доске, не в памяти:

**Звонок 1 — план.** Вход: бриф советника (problem-протокол) + root_spec +
реестр инструментов. Выход строго JSON:
```json
{
  "sources": [{"url": "...", "what": "...", "freshness_hint": "..."}],
  "need_tools": [{"tool_name": "...", "purpose": "...", "io": "..."}],
  "tool_runs": [{"tool": "...", "args": ["data/cars.json", "100"]}],
  "data_criteria": [{"cmd": "...", "expect": "..."}]
}
```
Харнесс (talk.js) превращает: need_tools → задачи type=tool (26.2);
tool_runs → задачи type=tool_run с data_criteria в criteria и deps на
соответствующий tool; координатор их доводит. Критерии данных — свой класс
(§10): свежесть (дата последней записи), объём (минимум N — правило
минимумов §8.3!), правдоподобие диапазонов; технически это обычные
`{cmd,expect}` по файлу данных — новый механизм не нужен, нужна дисциплина
в промпте researcher + architect.

**Звонок 2 — оценка.** Вход: отчёты tool_run (feedback задач) + первые
строки добытых данных (глаза read-only, фильтр мусора действует). Выход:
```json
{"verdict": "proceed", "data_summary": "...", "for_architect": "..."}
```
или
```json
{"verdict": "data_unavailable", "client_note": "этих данных не существует в открытом виде: ..."}
```
`data_unavailable` → проект `blocked_needs_human`, client_note — дословно
человеку. Это ЖЕСТОЧАЙШИЙ стоп-фактор §10: харнесс не имеет ветки
«продолжить без данных», её отсутствие — инвариант.

`proceed` → `for_architect` приклеивается к брифу, дальше обычный
project-поток. Отчёт консультанта (§17) обязан включить блок «данные взяты
оттуда-то, обновляются так-то» — источник: sources звонка 1 + манифесты
использованных инструментов.

### 6.2. Приём ТЗ-пакета (ingest)

`bin/ingest.js <папка>` (и команда `/ingest <папка>` в talk):
1. Читает текстовые форматы (md/txt/json; pdf и бинарные макеты — честная
   строка-заглушка в духе фильтра мусора §6.2: «[макет.pdf, 2МБ — не
   читается, учтён как артефакт]»; парсинг pdf — за рамками v1.0-final,
   зафиксировать).
2. По документу за вызов — advisor.spec-протокол (существующий
   prompts/advisor.spec.md, режим «разбор документа»): выжимка требований
   `[{id, text, source}]`.
3. Свод: единый root_spec (через существующий mergeRootSpec) + карта
   требований в root_spec.requirements + **поиск дыр и противоречий в самом
   ТЗ** (отдельный вызов advisor.spec: вход — все требования разом; выход —
   `{holes:[...], contradictions:[...], questions:[...]}`; вопросы клиенту —
   ТОЛЬКО по дырам, не по вкусам, §11).
4. Дыры печатаются ДО старта; ответы человека приклеиваются к root_spec;
   архитектор получает требования с ID и обязан вернуть трассировку
   `требование → задача` (поле `covers:[reqId]` в задачах DAG; требование
   без задачи = дыра, о которой сообщено).

### 6.3. Банк evals и judge DAG

- `evals/bank/` — четыре эталонных желания из Г§10: кофейня, танки
  (браузерная игра), аналитика откликов (Chart.js-класс), диллер-лайт
  (problem). Формат: `{input, route_expected, protocol_expected,
  min_tasks, forbidden_in_interview}`.
- `evals/bank.js` — прогон банка: в MOCK — маршрутизация+структура
  (быстрый смоук), с живыми моделями — полный (релизный критерий любых
  правок промптов, Г§10).
- `evals/judge.js` — LLM-as-judge (§14.3): после прогона architect'а один
  вызов `chat('cheap', DAG+рубрика, {json:true})` → `{atomicity, dupes,
  touches_realism, score}` → событие `eval_score`. Judge НИКОГДА не
  блокирует прогон — только метрика (§14: уровень 3 — наблюдение, не ворота).
