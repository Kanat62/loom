# LOOM — Руководство разработки
### Практический план сборки рабочего продукта. Пара к ТЗ v4: там — ЧТО и ПОЧЕМУ, здесь — КАК и В КАКОМ ПОРЯДКЕ.

Правила чтения: разделы идут в порядке сборки; ничего не строится «на будущее» —
каждый файл появляется в той фазе, где нужен; фаза не закрыта, пока не пройден её
реальный прогон (не MOCK). Ссылки вида §N — на разделы ТЗ v4.

где физически лежит код, который их вычисляет.

## 1. Ключевые контракты кода (не менять без правки ТЗ)

**Шлюз:** `chat(role, messages, {json}) → текст | распарсенный объект`.
Имя модели `claude-cli:tier` → `claude -p --model <tier-id> --output-format json
--system-prompt-file <tmp> --tools "" --strict-mcp-config --mcp-config mcp.empty.json
--settings settings.headless.json`, system — файлом, user — STDIN (шрамы 28–29).
Ошибка/таймаут/не-JSON = сбой модели → следующая в цепочке; два круга (§5).

**Журнал:** `addTask(t)`, `claimNext(role) → task|null` (BEGIN IMMEDIATE, фильтр
role + разблокированные deps), `setStatus(id, status, {feedback?, question?})`
(feedback перезаписывается, не копится), `releaseStuck()`, `logEvent({run_id,...})`.
run_id генерируется на входе запроса и прошивает все events (§14).

**Критерий:** JSON `{cmd, expect}`; исполняет только `runCheck(cmd,{cwd,timeout})`
— `node -e` вынимается в `.loom-check-*.cjs` внутри workspace (шрам 7).

**Кодер (вариант А):** вход = system(prompts/coder.md) + user(задача + feedback +
дельта файлов §6); выход строго `{"files":{...}}` или `{"question":"..."}`;
запись — harness'ом, только под workspace (замок пути).

**Router:** `{route, reason, criteria?}`; ошибка парсинга → route=project.

## 2. Фаза 0 — фундамент (день 1)

Файлы: package.json, .env, .gitignore, config.js, journal.js, skills.js, io.js, mock.js (пустые ветки).
- Каталог `~/dev/loom` (вне облачных синков, шрам 6). `git init` сразу.
- package.json: `"talk": "node --env-file=.env --no-warnings talk.js"` и аналогично
  cli/resume/ask/mock-варианты (инвариант 1).
- journal.js: адаптер — пробуем `better-sqlite3`, при ошибке сборки — `node:sqlite`
  (инвариант 8); схемы §4 целиком, включая events с run_id.
- config.js: ROLES из §5 (стартовая claude-cli-конфигурация), MODEL_LIMITATIONS={},
  MAX_ATTEMPTS=4, MODEL_TIMEOUT=300_000, WORKSPACE.

**DoD:** `node --env-file=.env -e "import('./journal.js').then(j=>console.log(j.addTask({title:'t',spec:'s',criteria:'{}'})))"`
создаёт запись; claimNext в двух параллельных процессах не выдаёт одну задачу дважды
(скрипт-проверка гонки — написать и оставить в evals/race.js).

## 3. Фаза 1 — один агент по-настоящему (дни 2–4)

Файлы: gateway.js (только claude-cli-ветка), agents/coder.js, agents/tester.js,
checkRunner.js, coordinator.js, cli.js, prompts/coder.md.
- Сначала руками: `claude -p "скажи ok" --output-format json` работает в терминале.
- gateway: тиры через `--model` (маппинг tier→ID моделей в config), usage → events.
- coder: глаза = ПОЛНЫЙ снимок (дельта — Фаза 2), фильтр мусора СРАЗУ (.min, >50КБ,
  правило CDN в промпте — шрам 30); замок пути.
- tester: runCheck, отчёты 1000 символов (шрам 20).
- coordinator: цикл §7-лестница без эскалации тиров (просто attempts), releaseStuck на старте.
- cli.js: задача задаётся руками в коде/аргументом БЕЗ советника — цель фазы: конвейер
  задача→код→проверка, не интервью.

**Реальный прогон (обязателен):** задача «sum.js: экспортирует sum(a,b); критерий
node -e require('./sum.js').sum(2,3) → 5» — done с первой-второй попытки.
**DoD:** прогон зелёный; в events есть run_id, usage, длительности; `node evals/race.js` зелёный.

## 4. Фаза 2 — Context Engineering (дни 5–6)

- currentFiles → дельта (снимок на ходе 2, дальше added/changed/removed; шрам 9).
- prompts/*.md статичны, динамика в user; проверка кэша: два вызова подряд →
  во втором cache_read > 0 (лог из usage).
- feedback = только последний (проверить UPDATE, не конкатенация).
- budget_usd: считать из usage (для claude-cli — условные цены тиров в config),
  писать spent_usd, стоп при превышении → blocked_needs_human.

**Реальный прогон:** та же задача + вторая с зависимостью; сравнить tokens_in хода 3
с Фазой 1 (ожидание: кратное падение). **DoD:** SQL `SELECT agent, avg(tokens_in) FROM events GROUP BY agent` показывает дельту; бюджет-стоп срабатывает на игрушечном лимите 0.01.

## 5. Фаза 3 — роли, Router, полный вход (дни 7–12)

Файлы: agents/advisor.js + prompts/advisor.*.md, agents/router.js + prompts/router.md,
agents/architect.js + prompts/architect.md, agents/skill.js, talk.js, resume.js, ask.js, engine.js.
- Router: 5 типов (§1); criteria для tweak; complexity_score; эскалация tweak→project.
- Advisor: выбор протокола по route; product/engineering split; **regex-фильтр кодом**
  (шрам 15); guard мусорного входа (шрам 16); engineering_defaults в бриф.
- Architect: prompts/architect.md = Свод §8 дословно; packages→npm install до кодера
  (шрам 22); предпроверка критериев с тихим логом (шрам 19); контракт регрессии §8.9
  (склейка существующих критериев, НЕ генерация новой проверки — шрам 31).
- talk.js: сеанс, /status /resume /exit, unhandledRejection-броня, .history-бэкап
  (tweak/project, не question), санитизация ввода (шрам 32).
- skill.js: фильтр галлюцинаций + SKIP (шрам 24).

**Реальные прогоны (все три маршрута):** «что делает sum.js?» → question, ноль записей;
«поменяй вывод calc на слово Итого:» → tweak, одна задача; «сделай одностраничник
кофейни: h1 'Кофе на Манаса', меню 5 позиций, контакты, тёмная тема» → project,
4–6 задач + регрессия, открывается в браузере.
**DoD:** лог маршрутов корректен; в интервью НЕТ технических вопросов (прогнать
evals/interview.js: 5 эталонных желаний → авто-проверка Haiku'ом «есть ли px/мс/алгоритмы»);
предпроверка печатает по строке на критерий; регрессия — повтор существующих критериев (видно в журнале).

## 6. Фаза 4 — безопасность (дни 13–15)

- docker/coder.Dockerfile: node:22-slim, COPY package*.json + npm ci на сборке;
  запуск кодер-исполнения критериев и (при варианте Б позже) самого кодера:
  `--network none --memory 2g --cpus 2 --pids-limit 256 -v workspace:/workspace`.
  В варианте А в Docker уезжает ИСПОЛНЕНИЕ (checkRunner + npm install), не вызов модели.
- gitGuard: pre-flight в coordinator — текущая ветка репо LOOM не main-агентская,
  агентские операции только в workspace (шрам 5).
- Заготовка tsc-контрактов: tsconfig + скрипт level-0 (включается в Фазе 6).

**DoD:** критерий с `rm -rf /` внутри workspace-контейнера не трогает хост;
попытка кодера записать `../evil.js` блокируется и логируется.

## 7. Фаза 5 — GitHub (дни 16–17)

gh CLI: репо продукта создаётся/пушится скриптом координатора при project-сдаче;
auto-commit safety net в конце задач кодера (шрам 4). **DoD:** project-прогон
заканчивается репозиторием с историей по задачам.

## 8. Фаза 5.5 — обживание и сдача (дни 18–20)

- agents/livein.js + prompts/livein.md: вход — root_spec + URL/файл продукта; проход
  Playwright'ом «глазами новичка» (открылось? понятно за 10с? главный сценарий?
  375px?); выход — JSON списка шероховатостей → tweak/polish-задачи с criteria.
  Лимит: один цикл на сдачу.
- Отчёт консультанта (§17): генерируется advisor_analyst из root_spec + доска +
  engineering_defaults + источники данных; печатается при закрытии project-дерева.

**Реальный прогон:** кофейня из Фазы 3 → live_in находит ≥1 реальную шероховатость →
tweak закрывается → отчёт консультанта читабелен.
**DoD:** в отчёте есть: проблема/цель, принятые решения с «почему», как проверено
(оба контура), что требует человека.

## 9. Фаза 6 — параллельность (когда однопоточно скучно-стабильно)

worktrees + CODER_WORKERS=N; mergeQueue (done→merge_pending→merge→rebase остальных);
touches_files/functions в задачах архитектора; tsc level-0 включён; fan-in политика;
LLM-reconciler (Haiku) на конфликтах. **DoD:** проект из 8+ задач в 2 воркера:
ни одного двойного захвата (events это докажут), очередь мёржей последовательна,
искусственный конфликт двух задач в одном файле разрешён без человека.

## 10. Фаза 7 — глубина входа (по готовности)

- ingest.js: папка с ТЗ/доками → advisor.spec-протокол → root_spec + карта
  требований → дыры клиенту до старта (§11).
- researcher.js + prompts/researcher.md: источники → заказ парсеров конвейеру →
  критерии данных (свежесть/объём/правдоподобие) → ЖЁСТКИЙ стоп «данных нет» (§10).
- evals/: банк желаний (кофейня, танки, аналитика, диллер-лайт) + interview-eval +
  judge DAG (§14.3). Прогон банка = релизный критерий любых правок промптов.

**Выпускной экзамен всей системы:** вход-problem «в КР растёт купля-продажа машин,
я диллер, не знаю какие заказывать» — LOOM сам доходит до «нужен не магазин, а
инструмент видеть рынок», честно отчитывается о доступности данных, строит то,
что построимо, сдаёт отчётом консультанта.

## 11. Правила разработки LOOM с помощью Claude Code

1. Шаблон промпта: контекст проекта (2–3 строки + «прочитай файлы X,Y») → проблема
   с историей ПОЧЕМУ (шрам/инцидент) → что сделать по пунктам → что НЕ трогать →
   проверка (всегда включает MOCK-прогон).
2. После ЛЮБОЙ правки машины: `npm run mock` (эталон: калькулятор, 2 задачи, первая
   с 2 попытками) — до реального запуска. Правка без зелёного mock не коммитится.
3. Один промпт = одна зона (шлюз ИЛИ советник ИЛИ координатор). Смешанные правки —
   источник регрессий, которые не отдиагностировать.
4. Диагностика багов — по правилу четырёх уровней (§19), И по events (SQL), не по
   памяти о логах.
5. Ревью диффа руками перед коммитом — всегда: Claude Code правит уверенно, в том
   числе уверенно неправильно.

## 13. Определение «реально рабочий проект» (release-критерий v1.0)

LOOM v1.0 считается рабочим, когда БЕЗ участия автора, на чистой машине по README:
(1) mock-прогон зелёный; (2) три маршрута отрабатывают на живых моделях; (3) проект
уровня «аналитика откликов» (5+ задач, Chart.js-класс) собирается end-to-end с
регрессией и обживанием ≤ чем за вечер; (4) убийство процесса в любой точке не
теряет работу (resume доводит); (5) отчёт консультанта на сдаче отвечает на «что,
почему, как проверено, что дальше»; (6) банк evals зелёный. Пункты (1)–(6) — и
только они — определяют готовность; всё остальное — развитие.
