# LOOM

Автономный инженер разработки — не исполнитель промптов. Берёт задачу (вопрос,
точечную правку, хотелку, проблему или пакет ТЗ), сам решает инженерные детали,
строит и проверяет продукт двумя независимыми контурами (код + впечатление
первого пользователя), сдаёт отчётом консультанта: что сделано, почему именно
так, как проверено, что требует вашего решения.

Полная спецификация — [`docs/LOOM_TZ_v4.md`](docs/LOOM_TZ_v4.md) (ЧТО и ПОЧЕМУ) и
[`docs/LOOM_DEV_GUIDE.md`](docs/LOOM_DEV_GUIDE.md) (КАК и в каком порядке). Решения,
принятые в неочевидных местах при сборке — [`docs/DECISIONS.md`](docs/DECISIONS.md).

## Что построено

**Часть 1** (см. `docs/LOOM_DEV_GUIDE.md`): фазы 0–4 и 5.5 — фундамент → один
агент → context engineering → роли и router → безопасность → обживание и
отчёт консультанта.

**Часть 2** (см. `docs/LOOM_DEV_GUIDE_PART2.md`, в процессе — статус построчно
в `docs/DECISIONS.md` под заголовком «Часть 2»): журнал-фундамент Кузницы
инструментов и настоящей параллельности (П§2 — таблицы `tools`/`task_deps`,
fan-in регрессии); Фаза 5 — публикация в приватный GitHub через `gh` CLI,
включается явным `GITHUB_PUSH=1`, по умолчанию выключено (П§3); **Фаза 5.7 —
Кузница инструментов (§26)**: LOOM сам собирает, регистрирует и переиспользует
свои инструменты (`core/tools.js`, `tools/<name>/` со своим git-репо,
`type='tool'`/`'tool_run'` задачи, reuse-first в контексте архитектора,
`npm run cli -- --type tool ...` для ручной сборки) — П§4; **Фаза 6 —
параллельность (§16)**: `CODER_WORKERS=N` в `.env` → N воркеров в одном
процессе, каждый в своём git worktree продукта (`core/worktrees.js`),
weave по `touches_files` в `claimNext` (не даёт двум воркерам одновременно
писать в один файл), последовательная очередь мёржей с LLM-reconciler на
конфликт (`core/merge.js`, роль `cheap`, выбор ours/theirs — не пишет
новый код) и tsc-ворота уровня 0 перед мёржем (опционально, если
`typescript` доступен локально) — П§5. **Фаза 7 — глубина входа**:
researcher (`agents/researcher.js`, §10) — двузвонковый цикл план→оценка
поверх Кузницы, жёсткий стоп-фактор `data_unavailable` (архитектор просто
не вызывается, ветки «продолжить без данных» в коде нет); приём ТЗ-пакета
(`agents/ingest.js` + `bin/ingest.js` + `/ingest <папка>` в talk, §11) —
разбор документов на требования с id, поиск дыр/противоречий ДО архитектора,
трассировка `covers:[reqId]` в задачах архитектора (непокрытое требование —
отчётность, не блокировка); банк эталонных желаний (`evals/bank/*.json` +
`evals/bank.js`, `npm run bank:mock`/`npm run bank`) и LLM-as-judge
(`evals/judge.js`, событие `eval_score`, метрика — никогда не ворота) — П§6.
**Фаза 8**: FTS5 для skills/tools (порог 100+ записей, `INSERT OR IGNORE`
backfill — `NOT IN`-вариант молча портит индекс внешне-контентной FTS5-
таблицы, найдено `evals/fts.js`, фолбэк в LIKE при сборке sqlite без FTS5)
— П§7 п.1; дашборд `bin/dash.js` (node:http на 127.0.0.1, ноль
зависимостей, `npm run dash`, три колонки: статусы/детали/чат-с-
аналитиком, read-only) — П§7 п.2; maintenance-минимум — `type=
regression_detected` уже маршрутизируется как tweak (без новой ветки
координатора), `AUTO_MERGES_PER_DAY` — реальный гейт в `drainMergeQueue`
(events за 24ч) — П§7 п.3; Telegram-вход `bin/telegram.js` (long-poll,
опционально, `npm run telegram`, включается `TELEGRAM_BOT_TOKEN`,
намеренно урезан — без researcher/ingest/live_in) — необязательный модуль.
Полный контур PR→CI→staging→прод (§18) — сознательно за рамками
v1.0-final.

**§29 — изоляция проектов** (реальный инцидент, шрам 35, см.
`docs/LOOM_TZ_v4_SECTION_29.md` и DECISIONS «§29»): проект — первоклассная
сущность (`core/projects.js`), не просто колонка `project_id` без фильтра.
`claimNext`/`runCoordinatorLoop` требуют `projectId` и падают без него
(fail-loud); каждый проект живёт в своём `<WORKSPACE>/<project_id>/` —
замок пути (§12.1) стал границей между проектами тем же механизмом, что уже
защищал от `../`. `npm run talk` — команды `/projects` (список со
счётчиками) и `/project <id|номер>` (переключить активный); новый
wish/problem/spec без активного проекта (или после явного «это новый
продукт») заводит свежий проект автоматически. Старая единая база
(`project_id='default'`) необратимо мигрирует в архивный `'legacy'` при
первом же старте после обновления — `/project legacy` открывает её для
ручного разбора.

## Быстрый старт

```bash
npm install
claude login              # если ещё не логинены в подписку Claude Code
cp .env.example .env      # секретов внутри нет, claude-cli работает по подписке
npm run mock               # детерминированный смоук без единого вызова модели
npm run talk                # интерактивная сессия: question/tweak/wish/problem/spec
```

Полный пошаговый прогон (включая реальные вызовы моделей, которые я по вашей
просьбе не запускал) — в [`docs/RUNBOOK.md`](docs/RUNBOOK.md).

## Структура

См. `docs/LOOM_DEV_GUIDE.md` §0 для полного дерева и §1 для контрактов кода.
Коротко: `core/` — машина без LLM-специфики ролей (`journal.js`/`skills.js` —
две SQLite-базы с фолбэком better-sqlite3→`node:sqlite`; `projects.js` —
реестр проектов (§29, таблица `projects` в journal.db); `gateway.js` —
единственная точка вызовов модели (claude-cli); `coordinator.js` —
детерминированный цикл захват→исполнение→вердикт, требует `projectId`;
`mock.js` — детерминированные ответы всех ролей для `MOCK=1`); `agents/*.js` — роли
(router, advisor, architect, coder, tester, skill, live_in, researcher,
ingest); `bin/` — точки входа (`talk.js`/`resume.js`/`ask.js`/`cli.js`/
`ingest.js`); `docs/` — спецификация и раннбук.

## MOCK-режим

`npm run mock` — калькулятор (2 задачи, первая с нарочной ошибкой первой
попытки — демонстрирует цикл поймал→отчёт→починил), полный конвейер
router→advisor→architect→coordinator→skill_writer→live_in→отчёт консультанта,
tweak, question и проверка замка пути; изоляция проектов (§29) — два
проекта, раздельный захват/root_spec/замок пути между ними, `claimNext` без
`projectId` честно бросает исключение, миграция `'default'→'legacy'`
идемпотентна. Ни одного вызова модели — обязательный смоук после любой
правки машины (см. `docs/LOOM_DEV_GUIDE.md §11`).
